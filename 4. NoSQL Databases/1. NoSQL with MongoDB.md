# NoSQL with MongoDB — Simple Summary

**NoSQL** is a broad term used for databases that are **not traditional relational (SQL) databases**. It doesn't describe one specific type of database—instead, it includes many different database models.

In this section, the focus is on **MongoDB**, which is a **document-based NoSQL database**.

Unlike relational databases, MongoDB stores data as **documents** (similar to JavaScript objects) and **does not require a fixed schema**. This makes it flexible and easy to use, especially for JavaScript developers.

## Key Points

* NoSQL means "not a traditional relational database."
* MongoDB is a **document database**.
* Documents are stored as JSON-like objects.
* No predefined schema is required.
* Different documents in the same collection can have different fields.
* MongoDB feels natural for JavaScript developers because documents look like JavaScript objects.

---

# What is a Document Database?

A document database stores information as **documents** instead of rows and columns.

Example document:

```json
{
  "name": "Luna",
  "type": "dog",
  "breed": "Havanese",
  "age": 14
}
```

Each document usually represents **one thing**, such as:

* One user
* One pet
* One product
* One blog post

---

# Why Choose MongoDB?

One of MongoDB's biggest advantages is its **flexible schema**.

In a relational database, you must define the structure before storing data.

Example schema:

```text
Users

Name
Age
City
```

Every row must follow that structure.

In MongoDB:

```json
{
  "name": "Brian"
}
```

Another document can be:

```json
{
  "name": "Alice",
  "city": "Seattle",
  "favoriteColor": "Blue"
}
```

Both are valid because MongoDB doesn't require every document to have the same fields.

## Advantages

* Very flexible
* Easy to change data structure
* Great for rapidly evolving applications
* Works naturally with JavaScript

## Disadvantages

Because MongoDB accepts almost any document, mistakes like misspelled field names can accidentally create new fields.

Example:

Correct:

```json
{
  "age": 25
}
```

Mistake:

```json
{
  "agge": 25
}
```

MongoDB accepts both, even though `agge` is likely a typo.

# MongoDB Structure

MongoDB organizes data like this:

```text
MongoDB Server
│
├── Database
│     ├── Collection
│     │      ├── Document
│     │      ├── Document
│     │      └── Document
│     └── Collection
```

---

# Databases

A **database** is a container for related collections.

Example:

```text
adoption
├── pets
├── owners
└── appointments
```

There are no strict rules about how to organize databases, but related collections are usually grouped together.

---

# Collections

A **collection** is similar to a table in a relational database.

A collection stores many documents.

Example:

```text
pets
├── Luna
├── Fido
├── Fluffy
└── Baxter
```

A good practice is:

* One collection = one type of data

Avoid mixing unrelated data (such as users and products) in the same collection.

Collections also support useful features such as:

* Indexes
* Capped collections (automatically discard old documents after reaching a size limit)

# Documents

A **document** is one JSON-like object.

Example:

```json
{
  "name": "Luna",

  "breed": "Havanese",
  "age": 14

```

Each document usually represents one real-world object.

## Start a MongoDB Docker Container

```bash
docker run --name test-mongo -dit -p 27017:27017 --rm mongo:8.2.5
```

### Explanation

* `docker run` → Creates and starts a new container.
* `--name test-mongo` → Names the container `test-mongo`.
* `-d` → Runs the container in **detached mode** (background).
* `-i` → Keeps STDIN open for interactive use.
* `-t` → Allocates a pseudo-terminal.
* `-p 27017:27017` → Maps port **27017** on the host to port **27017** in the container.
* `--rm` → Automatically removes the container when it stops.
* `mongo:8.2.5` → Uses the official MongoDB Docker image, version **8.2.5**.

### Command Flags

* `--name` → Assigns a custom container name.
* `-d` → Detached mode (run in background).
* `-i` → Interactive mode (keep input open).
* `-t` → Allocate a terminal.
* `-p` → Publish a container port to the host (`host_port:container_port`).
* `--rm` → Remove the container automatically after it exits.

---

## Open the MongoDB Shell

```bash
docker exec -it test-mongo mongosh
```

### Explanation

* `docker exec` → Runs a command inside an existing container.
* `-i` → Keeps STDIN open.
* `-t` → Allocates a terminal.
* `test-mongo` → The name of the running container.
* `mongosh` → Starts the MongoDB interactive shell.

### Command Flags

* `-i` → Interactive mode.
* `-t` → Allocate a terminal.

# Inserting Documents

MongoDB can automatically create databases and collections when data is inserted.

Example:

```javascript
db.pets.insertOne({
  name: "Luna",
  type: "dog",
  breed: "Havanese",
  age: 14
});
```

If the `pets` collection doesn't exist, MongoDB creates it automatically.

---

# Inserting Many Documents

You can insert multiple documents at once using `insertMany()`.

Example:

```javascript
db.pets.insertMany([
  {
    name: "Luna",
    type: "dog"
  },
  {
    name: "Fido",
    type: "dog"
  }
]);
```

The notes also generate **10,000 documents** automatically using JavaScript.

```js
db.pets.insertMany(
  Array.from({ length: 10000 }).map((_, index) => ({
    name: [
      "Luna",
      "Fido",
      "Fluffy",
      "Carina",
      "Spot",
      "Beethoven",
      "Baxter",
      "Dug",
      "Zero",
      "Santa's Little Helper",
      "Snoopy",
    ][index % 11],
    type: ["dog", "cat", "bird", "reptile"][index % 4],
    age: (index % 18) + 1,
    breed: [
      "Havanese",
      "Bichon Frise",
      "Beagle",
      "Cockatoo",
      "African Gray",
      "Tabby",
      "Iguana",
    ][index % 7],
    index: index,
  }))
);
```

The important idea isn't the JavaScript itself, but that `insertMany()` inserts multiple documents in one operation.

---

# Finding Many Documents

To retrieve many documents:

```javascript
db.pets.find();
```

MongoDB returns 20 documents from the collection.

---

# Finding Documents

To retrieve a single document:

```javascript
db.pets.findOne();
```

MongoDB returns one matching document from the collection.

---

# Counting Documents

MongoDB can count how many documents exist in a collection.

Example:

```javascript
db.pets.countDocuments();
```

If there are 10,000 pets, it returns:

```text
10000
```

---

# JavaScript Inside MongoDB

One interesting feature of MongoDB is that its shell understands JavaScript.

This allows you to generate data programmatically.

Example from the notes:

```javascript
Array.from({ length: 10000 }).map(...)
```

This creates an array containing 10,000 objects before inserting them into MongoDB.

# Relational Database vs MongoDB

| Relational Database   | MongoDB                  |
| --------------------- | ------------------------ |
| Stores data in tables | Stores data as documents |
| Fixed schema          | Flexible schema          |
| Rows and columns      | JSON-like objects        |
| Tables                | Collections              |
| Records               | Documents                |
| SQL queries           | JavaScript-style queries |

# One-Sentence Summary

**MongoDB is a document-based NoSQL database that stores flexible JSON-like documents inside collections, allowing developers to quickly store and retrieve data without defining a fixed schema in advance.**

The doc's docker command doesn't set up authentication at all — it just runs Mongo wide open (fine for local testing, bad for anything else). Here's how to add a user/password:

## Quick version: create an admin user

Once you're in `mongosh` (like the doc has you do), run:

```js
use admin

db.createUser({
  user: "admin",
  pwd: "yourStrongPassword",
  roles: [ { role: "root", db: "admin" } ]
})
```

`root` gives full superuser access — great for local dev, too much for production.

## Restart the container with auth enabled

Right now Mongo will let anyone connect without credentials, even after you create a user, unless you enable auth. Restart the container like this:

```bash
docker run --name test-mongo -dit -p 27017:27017 --rm \
  -e MONGO_INITDB_ROOT_USERNAME=admin \
  -e MONGO_INITDB_ROOT_PASSWORD=yourStrongPassword \
  mongo:8.2.5
```

The official image reads those two env vars on **first startup only** and automatically creates that root user for you — no manual `db.createUser()` needed. If the container already has data/state from a previous run, these env vars are ignored, so this only works cleanly on a fresh container.

Then connect with credentials:

```bash
docker exec -it test-mongo mongosh -u admin -p yourStrongPassword --authenticationDatabase admin
```

## A less-privileged user (better practice)

Once auth is on and you're logged in as admin, create scoped users per database/app instead of always using root:

```js
use adoption

db.createUser({
  user: "adoptionApp",
  pwd: "anotherPassword",
  roles: [ { role: "readWrite", db: "adoption" } ]
})
```

That user can only read/write the `adoption` database — can't touch other databases or do admin tasks. This is what you'd actually use in an application's connection string, e.g.:

```
mongodb://adoptionApp:anotherPassword@localhost:27017/adoption
```

A couple of notes: don't hardcode passwords in commands you'll keep in shell history or scripts — use env vars or a secrets manager. And `root` is meant for one-time setup/admin, not for your app's day-to-day connection.

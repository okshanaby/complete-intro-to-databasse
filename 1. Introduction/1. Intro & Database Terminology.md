# What is a Database and Common Database Terms — Simple Summary

A **database** is a place where an application stores its data so it can be used later. Instead of keeping data only in memory (which disappears when the server stops), the server saves it in a database. This allows servers to remain **stateless**, since the database remembers everything.

A database doesn't have to be a complex system—it could even be an Excel spreadsheet or a text file, though real applications usually use dedicated database software.

## Key Points

* A **database** stores data permanently.
* Databases keep an application's data even after the server restarts.
* Servers can remain **stateless** because the database stores the application's state.
* Examples of simple databases include:

  * Excel spreadsheets
  * Text files
  * SQL databases
  * NoSQL databases

## What is a Query?

A **query** is a command sent to a database asking it to perform an operation.

A query can:

* Retrieve data
* Insert new data
* Update existing data
* Delete data
* Calculate or summarize data (aggregation)

### Example SQL Queries

```sql
-- Get data
SELECT * FROM users;

-- Add data
INSERT INTO users (name, city)
VALUES ('Brian', 'Seattle');

-- Update data
UPDATE users
SET city = 'New York'
WHERE name = 'Brian';

-- Delete data
DELETE FROM users
WHERE name = 'Brian';
```

## What is a Schema?

A **schema** defines the structure (shape) of the data stored in a database.

Think of it like the column headers in an Excel spreadsheet.

Example JSON:

```json
{
  "name": "Brian",
  "city": "Seattle",
  "state": "WA"
}
```

The schema would be:

* name
* city
* state

## Schema Types

### Strict Schema

Some databases require the structure to be defined before storing data.

Example:

* PostgreSQL

Advantages:

* Better validation
* More consistency
* Fewer mistakes

### Flexible Schema

Some databases allow every document to have different fields.

Example:

* MongoDB

Advantages:

* Easier to change data structure
* More flexible during development

## Database vs Schema

| Database               | Schema                               |
| ---------------------- | ------------------------------------ |
| Stores the actual data | Defines how the data is organized    |
| Like an Excel workbook | Like the column names in a worksheet |

# Types of Databases — Simple Summary

There are many database types, each designed for different problems. This course mainly covers five major categories.

## 1. Relational Databases (SQL / RDBMS)

Data is stored in **tables** with rows and columns.

Examples:

* PostgreSQL
* MySQL
* SQLite

Best for:

* Banking
* Business systems
* Applications with structured data

Characteristics:

* Fixed schema
* Uses SQL
* Strong relationships between tables

---

## 2. Document Databases (NoSQL)

Data is stored as documents, usually JSON-like objects.

Example:

```json
{
  "name": "Brian",
  "city": "Seattle"
}
```

Examples:

* MongoDB
* CouchDB

Best for:

* Web applications
* Flexible data
* Rapid development

Characteristics:

* Flexible schema
* JSON documents
* Easy to evolve data structure

---

## 3. Graph Databases

Store data as **nodes** and **relationships**.

Example:

```text
Brian ----FRIEND---- Alice
   |
   |
WORKS_AT
   |
 OpenAI
```

Examples:

* Neo4j
* Amazon Neptune

Best for:

* Social networks
* Recommendation systems
* Relationship-heavy data

---

## 4. Key-Value Databases

Store information as simple **key → value** pairs.

Example:

```text
"user123" → {
  "name": "Brian",
  "age": 30
}
```

Examples:

* Redis
* DynamoDB

Best for:

* Caching
* Sessions
* Very fast lookups

---

## 5. Other Database Types

Although not covered in detail, it's useful to know they exist.

### Search Engines

Designed for fast full-text searching. 

Search engine databases like Solr are placed in front of primary databases to perform sophisticated full-text search operations. They allow search functionality to scale independently from the main database, preventing heavy search loads from affecting database performance. This is useful when you need advanced search features or want to isolate search traffic from your primary database workload.

Examples:

* Solr
* Sphinx

Used for:

* Website search
* Product search
* Document search

---

### Wide Column Databases

Similar to relational databases but more flexible. 

Wide column databases like Cassandra are designed for hyper-scaling scenarios. They feature multi-master (multi-primary) architecture where you can read and write to various individual nodes that then gossip with each other to spread information. This enables fast read/write latencies across multiple nodes, making them suitable for applications operating at massive scale.

Examples:

* Apache Cassandra
* Google Bigtable

Best for:

* Massive datasets
* High availability
* Large distributed systems

---

### Message Brokers

Instead of storing data, they pass messages between applications. 

Message brokers are systems where you throw messages onto a queue for later processing. They are useful when you need to handle many writes but can tolerate some delay before the writes are processed, or when you have events that need to be processed asynchronously. They can scale to process incredible amounts of data.

Examples:

* Apache Kafka
* RabbitMQ
* Celery

Used for:

* Event-driven systems
* Notifications
* Background processing

---

### Multi-Model Databases

Support multiple database models in one system.

Examples:

* Azure Cosmos DB
* SurrealDB
* ArangoDB

Some databases like MongoDB and PostgreSQL also provide features that allow them to behave as multiple database types.

# ACID — Simple Summary

**ACID** is a set of properties that make database operations reliable and safe.

It stands for:

* Atomicity
* Consistency
* Isolation
* Durability

Not every application needs full ACID guarantees. Stronger ACID protection usually means slower performance, so developers choose the right balance depending on the application.

## A — Atomicity

Everything in a transaction succeeds, or nothing does.

Imagine transferring money.

Wrong:

```text
Subtract $100
(Server crashes)
Add $100
```

Money disappears.

Correct:

```text
Subtract $100
Add $100
Commit transaction
```

If any step fails:

```text
Nothing changes.
```

This is called an **atomic transaction**.

---

## C — Consistency

The database should always remain in a valid state.

Example:

If the primary database crashes before changes reach backup servers, data can become inconsistent.

Consistency ensures every server eventually agrees on the same correct data.

---

## I — Isolation

Multiple users can use the database at the same time without interfering with each other.

Running two transactions simultaneously should produce the same final result as running them one after another.

---

## D — Durability

Once data is successfully saved, it should survive crashes or power failures.

Many databases write data to disk before confirming success.

Benefit:

* Data isn't lost after a crash.

Trade-off:

* Disk writes are slower than memory operations.

---

# ACID Trade-offs

ACID provides strong reliability, but it comes with performance costs.

Some operations absolutely require ACID, such as:

* Bank transfers
* Financial records
* Inventory management

Other operations may prioritize speed over strict guarantees, such as:

* Analytics
* Logging
* Temporary cache data

Many databases allow you to choose the level of reliability for individual queries or transactions.

## One-Sentence Summary

**A database stores an application's data permanently, a schema defines the structure of that data, queries are commands that interact with it, different database types solve different problems, and ACID provides rules that make database operations safe and reliable while balancing performance and consistency.**

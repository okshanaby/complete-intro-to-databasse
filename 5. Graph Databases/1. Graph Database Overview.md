# Graph Databases — Simple Summary

A **graph database** is a type of database designed to store and query **highly connected data**. Instead of organizing data into **tables** (like PostgreSQL) or **documents** (like MongoDB), it stores data as **nodes** (things) and **relationships** (connections).

Graph databases are best when the **relationships between data are just as important as the data itself**.

In these notes, the graph database used is **Neo4j**, one of the most popular graph databases.

---

# When Should You Use a Graph Database?

Graph databases are useful when data has many interconnected relationships.

Examples include:

* Social networks (friends, followers)
* Movies, actors, and directors
* Family trees
* Road maps and navigation
* Company organizational structures
* Recommendation systems (Netflix, Spotify)
* Fraud detection
* Knowledge graphs

For example:

* One movie has many actors.
* One actor appears in many movies.
* Some people are both actors and directors.

Trying to represent these relationships efficiently in relational databases can become complicated, while graph databases are designed specifically for this.

---

# Core Components of a Graph Database

A graph database has three main parts:

* Nodes
* Relationships (Edges)
* Properties

---

# Nodes (Entities)

A **node** represents a real-world object or thing.

Think of a node as:

* A person
* A movie
* A company
* A city
* A product

Example:

```text
(Person)
(Movie)
(Company)
```

Each node usually has one or more **labels** that describe what type of object it is.

Example:

```text
Person
Movie
Actor
Director
```

Someone like Angelina Jolie could have two labels:

```text
Person
Actor
Director
```

This tells Neo4j exactly what kind of node it is.

---

# Relationships (Edges)

Nodes become useful when they are connected together.

These connections are called **relationships** in Neo4j.

Other graph databases may call them **edges**.

Example:

```text
Tom Hanks ----ACTED_IN----> Forrest Gump
```

Here:

* Tom Hanks is a node.
* Forrest Gump is another node.
* ACTED_IN is the relationship connecting them.

---

# Relationship Direction

Every relationship in Neo4j has a direction.

Example:

```text
Alice ----MANAGED----> Bob
```

This means:

Alice manages Bob.

It does **not** automatically mean:

```text
Bob ----MANAGED----> Alice
```

Some relationships naturally have direction:

* MANAGED
* LOVES
* PARENT_OF
* CREATED

Others are effectively two-way:

```text
Alice ----WERE_PEERS---- Bob
```

Even though Neo4j stores a direction internally, your queries can often ignore it when direction doesn't matter.

---

# One-Way Relationships

Sometimes relationships only go one direction.

Example:

```text
Taylor ----LOVES----> Sue
```

This only says:

Taylor loves Sue.

It **does not** mean:

Sue loves Taylor.

If both people love each other, you need two relationships.

```text
Taylor ----LOVES----> Sue
Sue ----LOVES----> Taylor
```

---

# Self-Referential Relationships

A node can even connect to itself.

Example:

```text
Taylor ----LOVES----> Taylor
```

This is called a **self-referential relationship**.

Graph databases fully support this kind of connection.

---

# Properties

Nodes and relationships can store additional information called **properties**.

Properties are similar to:

* Columns in PostgreSQL
* Fields in MongoDB

Example Person node:

```text
Person
--------
Name: Alice
Age: 30
Location: London
```

Example Company node:

```text
Company
--------
Name: Google
Industry: Technology
Location: California
```

Relationships can also have properties.

Example:

```text
Alice ----WORKS_FOR----> Google

Since: 2022
Position: Software Engineer
```

Here, the relationship stores information about **when** Alice started working there and **what position** she has.

---

# Example Graph

Imagine this small graph:

```text
(Tom Hanks)
     |
 ACTED_IN
     |
     V
(Forrest Gump)
     ^
     |
DIRECTED
     |
(Robert Zemeckis)
```

There are three nodes:

* Tom Hanks
* Forrest Gump
* Robert Zemeckis

And two relationships:

* ACTED_IN
* DIRECTED

This makes it very easy to ask questions like:

* Which movies did Tom Hanks act in?
* Who directed Forrest Gump?
* Which actors worked with this director?

---

# Labels

Labels describe what type of node something is.

Example:

```text
Person
```

or

```text
Movie
```

or multiple labels:

```text
Person
Actor
Director
```

Labels make searching much faster because Neo4j knows what kind of object it's looking for.

---

# Why Graph Databases Are Useful

Graph databases make relationship queries much simpler.

Instead of joining many tables like in SQL, Neo4j directly follows connections between nodes.

This makes it very efficient for questions like:

* Friends of friends
* Shortest path between cities
* Recommendations
* Organizational hierarchies
* Network analysis

---

# Neo4j

**Neo4j** is one of the most popular graph databases.

It uses:

* Nodes
* Relationships
* Labels
* Properties

to model connected data naturally.

The concepts you learn in Neo4j also apply to many other graph databases.

---

# Other Graph Databases

Neo4j is not the only graph database.

Other graph databases include:

* Amazon Neptune
* ArangoDB
* TigerGraph
* OrientDB
* JanusGraph

Although their syntax differs, they all use similar concepts:

* Nodes
* Relationships (Edges)
* Properties

# Graph Databases vs PostgreSQL vs MongoDB

| PostgreSQL                    | MongoDB                          | Graph Database (Neo4j)                  |
| ----------------------------- | -------------------------------- | --------------------------------------- |
| Stores data in tables         | Stores JSON-like documents       | Stores nodes and relationships          |
| Excellent for structured data | Excellent for flexible documents | Excellent for highly connected data     |
| Uses SQL joins                | Embeds or references documents   | Directly traverses relationships        |
| General-purpose               | General-purpose                  | Specialized for relationship-heavy data |

---

# Should You Replace PostgreSQL or MongoDB?

Usually **no**.

Graph databases are **specialized databases**, not general-purpose ones.

A common architecture is:

```text
Application
     |
     +------ PostgreSQL
     |
     +------ Neo4j
```

or

```text
Application
     |
     +------ MongoDB
     |
     +------ Neo4j
```

In this setup:

* PostgreSQL or MongoDB stores the main application data.
* Neo4j stores and queries complex relationships.

This lets each database do what it does best.

---

# Key Points

* A graph database stores data as **nodes** and **relationships**.
* Nodes represent real-world objects (people, movies, companies, etc.).
* Relationships connect nodes together and always have a direction in Neo4j.
* Both nodes and relationships can have **properties**.
* Relationships can even connect a node to itself.
* Graph databases are ideal for highly connected data such as social networks, recommendations, and organizational structures.
* Neo4j is one of the most popular graph databases.
* Graph databases are specialized tools and are commonly used alongside PostgreSQL or MongoDB rather than replacing them.

# One-Sentence Summary

**A graph database like Neo4j stores data as nodes connected by relationships, making it ideal for applications where understanding and querying complex connections between data is more important than simply storing the data itself.**

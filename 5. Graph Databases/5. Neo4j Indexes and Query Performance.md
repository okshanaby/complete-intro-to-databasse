# Neo4j Indexes and Query Performance — Simple Summary

Just like **PostgreSQL** and **MongoDB**, Neo4j uses **indexes** to make frequently used queries faster.

Without an index, Neo4j may need to scan every node in the database to find matching data.

With an index, Neo4j can quickly locate the required nodes without checking everything.

Indexes are especially important for **hot paths** — queries that run frequently in your application.

---

# Example Problem: Finding People Born in a Specific Year

Imagine an application feature:

> "Find celebrities who were born in the same year as me."

The query might look like this:

```cypher id="z2gxq4"
MATCH (p:Person)
WHERE p.born = 1967
RETURN p;
```

### Result

Returns all `Person` nodes where:

```text id="2yk8vy"
born = 1967
```

Example:

```text id="w6m2wi"
Michael Cera
Anna Kendrick
...
```

---

## What this query does

The query:

1. Searches all nodes with the label `Person`.
2. Checks their `born` property.
3. Returns people born in 1967.

---

## The Performance Problem

Without an index, Neo4j must check every `Person` node.

Example:

```text id="d4d9y8"
Database:

Person 1 → born 1980
Person 2 → born 1967 ✓
Person 3 → born 1990
Person 4 → born 1975
...
Person 1,000,000 → born 1967 ✓
```

Neo4j has to inspect everything.

With a small database this is fine.

With a database like the entire IMDb dataset, this could become very slow.

---

# Using EXPLAIN to Analyze Queries

Neo4j provides the `EXPLAIN` command to show how it plans to execute a query.

Run:

```cypher id="zjhr7s"
EXPLAIN
MATCH (p:Person)
WHERE p.born = 1967
RETURN p;
```

### Result

Neo4j shows a query execution plan.

Example:

```text id="kq7xqs"
Node Label Scan
    |
    V
Filter born = 1967
    |
    V
Return results
```

It may show something like:

```text id="r8i8w4"
Scan 133 Person nodes
Filter down to 13 results
```

---

## What EXPLAIN Does

`EXPLAIN`:

* Does not execute the query.
* Does not change data.
* Only shows the planned execution strategy.

It helps you understand:

* Whether indexes are used.
* How many nodes are scanned.
* Where performance problems may exist.

---

# Creating an Index

Create an index on the `born` property:

```cypher id="o1xw6q"
CREATE INDEX FOR (p:Person)
ON (p.born);
```

### Result

Creates an index that maps:

```text
Person.born
```

to the matching nodes.

---

## What this query does

Before the index:

```text id="3tdlkn"
Search every Person node
        |
        V
Check born property
```

After the index:

```text id="0p7z9s"
Search index
      |
      V
Jump directly to matching Person nodes
```

---

## Breaking Down the Index Command

```cypher id="w0b9g5"
CREATE INDEX
```

Creates a new database index.

---

```cypher id="x2q8cb"
FOR (p:Person)
```

Specifies:

* Label:

```text id="f0s6iw"
Person
```

The index applies to nodes with this label.

---

```cypher id="c42j9k"
ON (p.born)
```

Specifies the property being indexed.

The index stores:

```text id="1t8e8y"
Person.born
```

Run:


### Result


### Result

Returns people born in 1967.

Now Neo4j can find them more efficiently.

# EXPLAIN vs Running the Query

## EXPLAIN

```cypher id="8r7gwp"
EXPLAIN MATCH ...
```

Only shows the execution plan.

It does:

* Not run the query.
* Not return data.

---

## Normal Query

```cypher id="v2ek8r"
MATCH ...
RETURN ...
```

Actually executes the query and returns results.

# Viewing Existing Indexes

To see all indexes in Neo4j:

```cypher id="xq3z8k"
SHOW INDEXES;
```

### Result

Example:

```text id="h88a8r"
name        type        entityType   properties
----------  ----------  -----------  ----------
born_index  RANGE       NODE         born
```

---

## What this query does

Displays information about all indexes in the database.

You can see:

* Index name.
* Index type.
* What labels it applies to.
* Which properties are indexed.

# Query Performance Concepts

## Without Index

Query:

```cypher id="jzv1r0"
MATCH (p:Person)
WHERE p.born = 1967
RETURN p;
```

Execution:

```text id="1f3q8s"
Scan every Person node
        |
        V
Check born property
        |
        V
Return matches
```

Complexity increases as your database grows.

---

## With Index

Execution:

```text id="w7q0me"
Search index
      |
      V
Find matching nodes
      |
      V
Return results
```

Much faster for large datasets.

---

# When Should You Create Indexes?

Good candidates:

* Properties frequently searched.

Examples:

```text id="4n1k1m"
Person.name
Person.born
Movie.title
User.email
Product.id
```

---

Avoid creating indexes on everything.

Indexes:

* Improve reads.
* Require storage space.
* Slightly slow down writes because the index must also be updated.

---

# Key Points

* Indexes improve query performance by allowing Neo4j to find data faster.
* Without indexes, Neo4j may scan every matching node.
* `EXPLAIN` shows how Neo4j plans to execute a query.
* `CREATE INDEX` creates an index for faster lookups.
* `SHOW INDEXES` lists all existing indexes.
* Indexes are most useful for frequently executed queries.
* Large databases benefit significantly from proper indexing.

# One-Sentence Summary

**Neo4j indexes improve query performance by creating fast lookup structures for frequently searched properties, preventing expensive full graph scans and making large-scale graph queries much faster.**

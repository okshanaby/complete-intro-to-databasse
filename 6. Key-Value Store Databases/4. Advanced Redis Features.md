# Advanced Redis Features — Simple Summary

Redis has many advanced features beyond basic key-value storage. Most applications only use commands like `SET`, `GET`, `INCR`, `HSET`, etc.

The features below are designed for more complex or high-performance applications.

Topics covered:

* Lua scripting (`EVAL`)
* Publish/Subscribe (Pub/Sub)
* Transactions
* LRU (Least Recently Used) Cache

---

# Lua Scripting (`EVAL`)

Redis includes a built-in **Lua interpreter**, allowing you to execute multiple Redis commands inside a single script.

This is useful when you need:

* Multiple operations performed together
* More complex logic than a single Redis command provides

However, most application logic should remain in your application code rather than inside Redis.

---

## Example 1 — Creating Multiple Keys

```redis
EVAL "for i = 0,9,1 do redis.call('SET', 'lua_key_' .. i, i * 5) end" 0
```

### Result

Creates the following keys:

```text
lua_key_0 = 0
lua_key_1 = 5
lua_key_2 = 10
...
lua_key_9 = 45
```

### What it does

The Lua script loops from **0** to **9**.

For every number:

* Creates a key
* Stores a calculated value

Equivalent keys:

```text
lua_key_0
lua_key_1
lua_key_2
...
lua_key_9
```

---

### Command Breakdown

`EVAL`

Executes a Lua script inside Redis.

Syntax:

```text
EVAL "<lua_script>" number_of_keys [keys...] [arguments...]
```

In this example:

* `"for i = 0,9,1 ..."` → Lua script
* `0` → No Redis keys are passed to the script

---

### Why is there a `0`?

Redis wants to know beforehand which keys the script will access.

This is important when using **Redis Cluster**, where data is split across multiple servers (called **sharding**).

Since this script doesn't explicitly provide any keys:

```text
0
```

means:

> "This script is not receiving any known Redis keys."

This works fine on a single Redis server but isn't compatible with Redis Cluster for this particular script.

---

# Example 2 — Using KEYS and ARGV

First create a key:

```redis
SET lua_key_name cool_stuff_
```

### Result

```text
OK
```

---

Run the Lua script:

```redis
EVAL "for i = 0,9,1 do redis.call('SET', redis.call('GET', KEYS[1]) .. i, i * ARGV[1]) end" 1 lua_key_name 100
```

### Result

Creates:

```text
cool_stuff_0 = 0
cool_stuff_1 = 100
cool_stuff_2 = 200
...
cool_stuff_9 = 900
```

---

### What it does

The script:

1. Reads the value stored in:

```text
lua_key_name
```

which contains:

```text
cool_stuff_
```

2. Appends the loop number.

3. Stores values multiplied by **100**.

---

### Command Breakdown

`1`

Means:

> One Redis key is being passed.

Redis key:

```text
lua_key_name
```

Argument:

```text
100
```

Inside Lua:

```text
KEYS[1]
```

is

```text
lua_key_name
```

and

```text
ARGV[1]
```

is

```text
100
```

---

### Important Note

Lua arrays begin at **1**, not **0**.

So:

```text
KEYS[1]
ARGV[1]
```

are the first items.

# Loading Lua Scripts

Instead of writing long scripts directly inside `EVAL`, Redis also lets you:

* Save scripts
* Execute them by SHA hash
* Reuse them efficiently

This is mostly used in larger production systems.

# Publish / Subscribe (Pub/Sub)

Redis supports **Publish/Subscribe messaging**.

Instead of repeatedly checking for updates (**polling**), applications can wait for Redis to notify them when something happens.

Example workflow:

```text
Publisher
      │
      ▼
    Redis
      │
      ▼
Subscribers
```

When a message is published:

* Every subscribed application immediately receives it.

---

## Common Uses

* Live chat
* Notifications
* Real-time dashboards
* Multiplayer games
* Event-driven systems

Instead of constantly asking:

> "Has anything changed?"

your application waits until Redis sends a notification.

---

# Transactions

Sometimes several Redis commands must succeed together.

For example:

* Update one key
* Delete another
* Increment a counter

If one operation fails, you don't want only half the work completed.

Redis transactions guarantee:

* Every command runs successfully
* Or none of them run

This is called an **atomic transaction**.

Nothing else can interrupt the commands while they execute.

---

## Transaction Commands

Redis provides these commands:

| Command   | Purpose                                   |
| --------- | ----------------------------------------- |
| `MULTI`   | Starts a transaction                      |
| `EXEC`    | Executes all queued commands              |
| `DISCARD` | Cancels the transaction                   |
| `WATCH`   | Watches keys for changes before executing |

These are mainly used when multiple operations must be treated as one unit.

# LRU (Least Recently Used)

Redis can automatically remove old data when memory becomes full.

This is called **LRU**:

> Least Recently Used

The idea:

* Frequently used data stays.
* Rarely used data is removed.

---

## Example

Imagine a website serving profile pictures.

Without Redis:

```text
User
   │
   ▼
Database
```

Every request hits the database.

---

With Redis:

```text
User
   │
   ▼
Redis Cache
   │
   ├── Found → Return image
   │
   └── Not Found
          │
          ▼
      Database
          │
          ▼
 Store in Redis
```

Workflow:

1. User requests a profile picture.
2. Check Redis.
3. If found, return it immediately.
4. If not found:

   * Read from the database.
   * Store it in Redis.
   * Return it to the user.

---

## When Redis Runs Out of Memory

Suppose Redis can only store:

```text
1 GB
```

Eventually it becomes full.

Instead of failing, Redis removes:

> The least recently used key.

Frequently requested images remain cached.

Rarely requested images are removed.

---

## Why This Is Useful

Imagine:

* 100 million users
* Only 10 million active users

Keeping every profile picture in memory would waste RAM.

LRU automatically keeps only the most useful data.

This greatly increases:

* Cache hits
* Application speed
* Memory efficiency

---

# Redis vs Memcached

**Memcached** is another in-memory caching system.

If your application only needs a simple cache:

* Store data
* Retrieve data
* Automatically remove old entries

Memcached is often faster because that's its only purpose.

Redis offers many additional capabilities:

* Strings
* Lists
* Hashes
* Sets
* Sorted Sets
* Pub/Sub
* Transactions
* Lua scripting
* Persistence
* Multiple eviction policies

So while Memcached focuses purely on caching, Redis is a much more feature-rich in-memory data store.

---

# Key Points

* **Lua (`EVAL`)** lets Redis execute multiple commands inside a Lua script.
* `KEYS` and `ARGV` allow Redis keys and additional arguments to be passed into Lua scripts.
* **Pub/Sub** enables real-time communication where applications react to published messages instead of polling.
* **Transactions** ensure multiple Redis commands execute atomically—either all succeed or none do.
* **LRU (Least Recently Used)** automatically removes the least recently accessed data when Redis reaches its memory limit.
* Redis can function as a powerful cache while offering many advanced features beyond what simple caching systems provide.

## One-Sentence Summary

**Redis includes advanced features such as Lua scripting, Publish/Subscribe messaging, atomic transactions, and automatic LRU cache eviction, making it suitable for building high-performance, real-time, and scalable applications beyond simple key-value storage.**

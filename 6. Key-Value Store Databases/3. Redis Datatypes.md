# Redis Data Types — Simple Summary

Redis supports several built-in data types. Choosing the right one depends on how you want to store and retrieve your data.

The main data types covered here are:

* Strings
* Lists
* Hashes
* Sets
* Sorted Sets

There are more advanced types available, but these are the most commonly used.

---

# String

A **String** is the simplest Redis data type.

It stores:

* Text
* Numbers (numbers are internally stored as strings)
* Binary data (such as images or files)

Example:

```redis
SET name Brian
TYPE name

SET visits 10
TYPE visits
```

### Result

```text
OK
string

OK
string
```

### What it does

Creates two keys:

* `name` → `"Brian"`
* `visits` → `"10"`

Then `TYPE` shows what kind of Redis data type they are.

Although `10` looks like a number, Redis stores it as a **string**.

### Command Breakdown

`SET`

* Creates or updates a key.

`TYPE`

* Returns the Redis data type of a key.

---

# Lists

A **List** is an ordered collection of values.

Think of it like:

* A JavaScript array
* A queue
* A stack

Lists are useful for:

* Notifications
* Recent activity
* Task queues
* Messages

---

## RPUSH

Add items to the end of a list.

```redis
RPUSH notifications:btholt "Call your mom" "Feed the dog" "Take out trash"
LRANGE notifications:btholt 0 -1
```

### Result

```text
(integer) 3

1) "Call your mom"
2) "Feed the dog"
3) "Take out trash"
```

### What it does

Creates a notification list and appends three notifications.

Then retrieves every notification.

### Command Breakdown

`RPUSH`

* Adds one or more items to the **right (end)** of a list.

`LRANGE`

Returns items from a list.

Parameters:

* `0` → first item
* `-1` → last item

So:

```text
0 -1
```

means:

> Return the entire list.

---

## Reading Part of a List

Example:

```redis
LRANGE notifications:btholt 1 2
```

### Result

```text
1) "Feed the dog"
2) "Take out trash"
```

Starts from index **1** and ends at index **2**.

Negative indexes count from the end:

| Index | Meaning     |
| ----- | ----------- |
| `-1`  | Last item   |
| `-2`  | Second last |
| `-3`  | Third last  |

---

## RPOP

Remove the last item.

```redis
RPOP notifications:btholt
```

### Result

```text
"Take out trash"
```

### What it does

Removes and returns the last notification.

---

## LPOP

Remove the first item.

```redis
LPOP notifications:btholt
```

### Result

```text
"Call your mom"
```

### What it does

Removes the oldest notification.

---

## LPUSH

Insert at the beginning.

```redis
LPUSH notifications:btholt "New notification"
```

### Result

```text
(integer) 2
```

### What it does

Adds the notification to the **front** of the list.

---

## LTRIM

Limit the size of a list.

```redis
LTRIM notifications:btholt 0 5
```

### Result

```text
OK
```

### What it does

Keeps only elements from index **0** through **5**.

Everything after index **5** is deleted.

Useful for keeping only the latest notifications or recent history.

---

# Hashes

A **Hash** stores multiple fields under one key.

Think of it like:

* A JavaScript object
* A Python dictionary

Instead of creating many keys:

```text
user:name
user:city
user:company
```

you can store everything together.

Example:

```redis
HSET btholt:profile title "principal program manager" company "Microsoft" city "Seattle" state "WA"

HGET btholt:profile city

HGETALL btholt:profile
```

### Result

```text
OK

"Seattle"

1) "title"
2) "principal program manager"
3) "company"
4) "Microsoft"
5) "city"
6) "Seattle"
7) "state"
8) "WA"
```

### What it does

Creates one profile containing multiple fields.

Instead of many Redis keys, everything belongs to:

```text
btholt:profile
```

### Command Breakdown

`HSET`

Stores field-value pairs inside a hash.

`HGET`

Retrieves one field.

`HGETALL`

Returns every field and value.

---

## Numeric Operations on Hashes

Hashes also support numeric commands.

Example:

```text
HINCRBY
```

This increases a numeric field inside a hash, similar to how `INCRBY` works for strings.

---

# Sets

A **Set** is an unordered collection of unique values.

Characteristics:

* No duplicates
* No guaranteed order

Useful for:

* User roles
* Tags
* Categories
* Permissions
* Unique IDs

Example:

```redis
SADD colors red blue green yellow

SMEMBERS colors

SISMEMBER colors green

SISMEMBER colors gold

SPOP colors
```

### Example Result

```text
(integer) 4

1) "red"
2) "green"
3) "blue"
4) "yellow"

(integer) 1

(integer) 0

"green"
```

### What it does

Adds four unique colors.

Checks whether certain colors exist.

Finally removes a random color.

### Command Breakdown

`SADD`

Adds values to a set.

Duplicate values are ignored.

`SMEMBERS`

Returns every member.

`SISMEMBER`

Checks whether a value exists.

Returns:

* `1` → true
* `0` → false

`SPOP`

Removes and returns a random member.

---

# Sorted Sets

A **Sorted Set** is similar to a normal set, but every value has a **score**.

Redis automatically sorts members by their scores.

Useful for:

* Leaderboards
* Rankings
* Priorities
* Top scores

Example:

```redis
ZADD ordinals 3 third
ZADD ordinals 1 first
ZADD ordinals 2 second
ZADD ordinals 10 tenth

ZRANGE ordinals 0 -1
```

### Result

```text
1) "first"
2) "second"
3) "third"
4) "tenth"
```

### What it does

Each item receives a score.

| Score | Value  |
| ----- | ------ |
| 1     | first  |
| 2     | second |
| 3     | third  |
| 10    | tenth  |

Redis automatically keeps everything sorted.

### Command Breakdown

`ZADD`

Adds a member with a score.

Syntax:

```text
ZADD key score value
```

`ZRANGE`

Returns members in sorted order.

Parameters:

* `0` → first item
* `-1` → last item

---

# JSON

Redis also supports a JSON data type through an additional module.

However, many applications don't need it.

Often developers simply store JSON as a normal string.

Example:

```text
{"name":"Brian","age":35}
```

can be stored using:

```redis
SET user:brian "{\"name\":\"Brian\",\"age\":35}"
```

Then retrieve it with:

```redis
GET user:brian
```

This is usually enough when Redis is only acting as a cache.

If you need to update or query individual JSON properties directly inside Redis, the JSON type can be useful, but many real-world applications instead use hashes or store the JSON string as-is.

---

# Choosing the Right Data Type

| Data Type  | Best Used For                               |
| ---------- | ------------------------------------------- |
| String     | Text, numbers, cache values, binary data    |
| List       | Notifications, queues, recent activity      |
| Hash       | User profiles, objects, grouped fields      |
| Set        | Unique collections like tags or permissions |
| Sorted Set | Rankings, leaderboards, priority queues     |
| JSON       | Complex JSON documents (less commonly used) |

---

# Key Points

* **Strings** are the simplest and most commonly used Redis data type.
* **Lists** store ordered values and support adding/removing items from either end.
* **Hashes** group multiple fields under one key, making them ideal for storing objects.
* **Sets** store unique, unordered values and prevent duplicates automatically.
* **Sorted Sets** assign scores to values, allowing Redis to keep them sorted automatically.
* JSON support exists but is often unnecessary because hashes and strings handle many common use cases.

## One-Sentence Summary

**Redis provides several data types—Strings, Lists, Hashes, Sets, and Sorted Sets—each designed for different storage patterns, allowing you to efficiently model simple values, collections, objects, unique groups, and ranked data.**

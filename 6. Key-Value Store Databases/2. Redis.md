# Redis Basics — Simple Summary

**Redis** is an in-memory key-value database. Instead of storing data in tables (PostgreSQL) or collections (MongoDB), Redis stores everything as **keys** and **values**.

It is commonly used for:

* Caching
* Session storage
* Counters
* Leaderboards
* Queues
* Temporary data

---

# Running Redis with Docker

Start a Redis container:

```bash
docker run -dit --rm --name=my-redis -p 6379:6379 redis:8.6-trixie
```

### Result

* Downloads the Redis image (if needed).
* Starts a Redis server in the background.
* Redis listens on port **6379**.

### Command Breakdown

* `docker run` — Creates and starts a new container.
* `-d` — Runs the container in the background (detached mode).
* `-i` — Keeps standard input open.
* `-t` — Allocates a terminal.
* `--rm` — Automatically removes the container when it stops.
* `--name=my-redis` — Names the container **my-redis**.
* `-p 6379:6379` — Maps your computer's port **6379** to Redis's port **6379**.
* `redis:8.6-trixie` — The Redis Docker image and version.

---

Open the Redis command-line interface:

```bash
docker exec -it my-redis redis-cli
```

### Result

You enter the Redis interactive shell.

Example prompt:

```text
127.0.0.1:6379>
```

### Command Breakdown

* `docker exec` — Runs a command inside a running container.
* `-i` — Keeps input open.
* `-t` — Opens an interactive terminal.
* `my-redis` — Container name.
* `redis-cli` — Redis command-line client.

# Redis Stores Data as Keys and Values

The most common Redis operations are **SET** and **GET**.

```redis
SET name "Brian Holt"
GET name
```

### Result

```text
OK
"Brian Holt"
```

### What it does

* `SET` stores a value.
* `GET` retrieves a value.

### Command Breakdown

`SET`

* First argument → key (`name`)
* Second argument → value (`Brian Holt`)

`GET`

* Takes only the key.
* Returns the stored value.

---

# Redis Key Naming

Unlike PostgreSQL or MongoDB, Redis has:

* No tables
* No collections

Everything exists in one large key space.

To keep data organized, use **namespaces** with the `:` separator.

Example:

```text
user:btholt
user:sarah_edo
user:scotups

payment:scotups
payment:1marc

user:address:btholt
```

## Why use `:`?

It makes keys organized and predictable.

Instead of:

```text
john
mary
address
payment
```

Use:

```text
user:john
user:mary
payment:john
payment:mary
```

This avoids key collisions.

### Best Practice

Always use `:` as the separator.

Example:

```text
user:btholt
product:15
order:42
```

Avoid using `/`.

---

# Binary-safe Keys

Redis keys are **binary-safe**, meaning almost anything can be a key—even binary data like images.

Example (not recommended):

```text
<JPEG image bytes>
```

Although possible, long binary keys slow comparisons.

A better approach is storing something like:

```text
image:9d3775ac...
```

using a hash (such as an MD5 hash) as the key.

# INCR and DECR

Redis can quickly increase or decrease numbers.

```redis
SET visits 0
INCR visits
INCR visits
INCR visits
DECR visits
```

### Result

```text
OK
(integer) 1
(integer) 2
(integer) 3
(integer) 2
```

### What it does

Creates a counter for page visits.

Every visitor:

```text
INCR visits
```

No need to first read the value.

### Commands

`INCR`

* Adds **1**.

`DECR`

* Subtracts **1**.

---

# INCRBY and DECRBY

Increase or decrease by any amount.

```redis
INCRBY score:seahawks 6
DECRBY score:broncos 3
```

### Result

```text
(integer) 6
(integer) -3
```

### What it does

Useful when changing values by more than one.

Examples:

* Game scores
* Points
* Likes
* Inventory

### Commands

`INCRBY`

* Adds the specified number.

`DECRBY`

* Subtracts the specified number.

---

# MSET and MGET

Store or retrieve multiple values at once.

```redis
MSET score:seahawks 43 score:broncos 8
MGET score:seahawks score:broncos
```

### Result

```text
OK

1) "43"
2) "8"
```

### What it does

Instead of multiple `SET` or `GET` commands, Redis performs them together.

Useful when reading or writing many keys.

### Commands

`MSET`

* Stores multiple key-value pairs.

`MGET`

* Retrieves multiple values.

# EXISTS

Checks whether a key already exists.

```redis
SET plane:0:0 visited
EXISTS plane:1:0
SET plane:1:0 visited
EXISTS plane:0:0
EXISTS plane:1:1
SET plane:1:1 visited
```

### Example Result

```text
OK
(integer) 0
OK
(integer) 1
(integer) 0
OK
```

### What it does

Useful for checking whether data is already stored.

Returns:

* `(integer) 1` → exists
* `(integer) 0` → does not exist

Example use:

* Tracking visited locations
* Preventing duplicate work
* Checking cache entries

---

# DEL

Deletes a key.

```redis
SET greeting hello
EXISTS greeting
DEL greeting
EXISTS greeting
```

### Result

```text
OK
(integer) 1
(integer) 1
(integer) 0
```

### What it does

* Stores a key.
* Confirms it exists.
* Deletes it.
* Confirms it no longer exists.

### Command

`DEL`

Removes one or more keys from Redis.

# NX and XX

These options control when `SET` succeeds.

## Example

```redis
SET color blue XX
SET color blue NX
SET color blue XX
SET color blue NX
```

### Result

```text
(nil)
OK
OK
(nil)
```

### Meaning

`NX`

* Only succeeds if the key **does not exist**.

`XX`

* Only succeeds if the key **already exists**.

Think of it as:

| Option | Meaning            |
| ------ | ------------------ |
| `NX`   | Set if Not eXists  |
| `XX`   | Set only if eXists |

---

## Using NX to Track Visited Locations

```redis
SET plane:0:0 visited NX
SET plane:1:0 visited NX
SET plane:0:0 visited NX
SET plane:1:1 visited NX
```

### Result

```text
OK
OK
(nil)
OK
```

### What it does

If Redis returns:

* `OK` → first visit
* `(nil)` → already visited

No need for an `EXISTS` check first.

# TTL (Time To Live)

TTL lets Redis automatically delete data after a certain amount of time.

This is one of Redis's most important features for caching.

Example:

```redis
SET fitness:total:btholt 750kj EX 3600
```

### Result

```text
OK
```

### What it does

Stores:

```text
fitness:total:btholt = 750kj
```

Then automatically deletes it after **3600 seconds (1 hour)**.

### Command Breakdown

`SET`

Stores a key-value pair.

`EX`

* Sets an expiration time.
* Value is in **seconds**.

`3600`

* One hour.

---

## Milliseconds

Redis also supports:

```text
PX
```

which works the same way but uses **milliseconds** instead of seconds.

---

# TTL Example

```redis
SET test_expire hi EX 5
EXISTS test_expire
```

### Immediate Result

```text
OK
(integer) 1
```

Wait 5 seconds:

```redis
EXISTS test_expire
```

### Result

```text
(integer) 0
```

The key automatically disappeared.

# The Thundering Herd Problem

Imagine:

* A cached value expires.
* Suddenly **1,000 users** request it at the same time.

Because the cache is gone:

* All 1,000 requests recalculate the expensive data.
* The server becomes overloaded.

This is called the **Thundering Herd Problem**.

A common solution is:

* Users only read from the cache.
* A background process refreshes the cache before it expires.
* This prevents many users from triggering the same expensive computation simultaneously.

# Key Points

* Redis stores data as **key-value pairs**.
* `SET` stores data.
* `GET` retrieves data.
* Use `:` to organize keys into namespaces.
* `INCR`, `DECR`, `INCRBY`, and `DECRBY` provide fast numeric updates.
* `MSET` and `MGET` work with multiple keys at once.
* `EXISTS` checks whether a key exists.
* `DEL` removes keys.
* `NX` sets a value only if the key does not exist.
* `XX` sets a value only if the key already exists.
* `TTL` (`EX` or `PX`) automatically expires keys, making Redis ideal for caching.
* Be aware of the **Thundering Herd Problem**, where many requests regenerate the same expired cache simultaneously.

## One-Sentence Summary

**Redis is a fast in-memory key-value database where data is organized using keys, supports efficient operations like counters and bulk reads/writes, and provides automatic key expiration (TTL), making it an excellent choice for caching and temporary data storage.**

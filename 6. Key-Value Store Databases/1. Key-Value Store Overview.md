# Key-Value Store Overview — Simple Summary

A **key-value store** is a type of database that stores data as **key → value** pairs.

Unlike relational databases (PostgreSQL) or document databases (MongoDB), key-value stores are designed for **very fast reads and writes**, not complex queries.

Most applications **do not use a key-value store as their main database**. Instead, they use it alongside another database to improve performance.

Example architecture:

```text
Application
     |
     |----> PostgreSQL (Main Database)
     |
     |----> Redis / Valkey (Cache)
```

---

# Key Idea

Think of a key-value store like a giant JavaScript object or dictionary.

Example:

```text
"user:1"  -> "Alice"
"user:2"  -> "Bob"
"count"   -> 150
```

You simply:

* Store a value using a key.
* Retrieve the value later using the same key.

Example:

```text
SET "user:1" = "Alice"
GET "user:1"

Result:
Alice
```

There are no joins, complex filters, or advanced queries like in SQL.

---

# Characteristics of Key-Value Stores

## Simple API

Most key-value stores mainly support operations like:

* Store a value.
* Retrieve a value.
* Delete a value.
* Update a value.

Example operations:

```text
SET key value
GET key
DEL key
```

Because the API is simple:

* Easy to learn
* Easy to use
* Very fast

---

## Extremely Fast

Since they don't perform expensive operations like:

* JOINs
* GROUP BY
* Complex filtering

they can handle **very large amounts of traffic**.

Example:

Instead of searching many tables:

```sql
SELECT *
FROM orders
JOIN customers
JOIN payments
```

A key-value store simply does:

```text
GET latest_orders
```

which is much faster.

---

# Why Not Use It as Your Main Database?

Key-value stores cannot easily answer questions like:

* Which customers spent over $500?
* Show all users created this month.
* Join customers with their orders.

Those are tasks for relational databases like PostgreSQL.

Instead:

```text
PostgreSQL
    ↓
Runs expensive query

↓

Redis stores the result

↓

Application reads from Redis
```

---

# Common Use Cases

## 1. Caching (Most Common)

Caching means storing frequently used data so it doesn't need to be recomputed or fetched repeatedly.

Example:

Without cache:

```text
User requests dashboard

↓

PostgreSQL runs a query

↓

5 seconds

↓

Result returned
```

Every user waits 5 seconds.

---

With cache:

```text
First request

↓

PostgreSQL query

↓

Store result in Redis

↓

Future requests

↓

Redis returns result

↓

Milliseconds
```

Benefits:

* Faster applications.
* Lower database load.
* Lower server costs.

---

### Example

Suppose this SQL query takes 5 seconds:

```sql
SELECT *
FROM yearly_sales;
```

Instead of running it every request:

```text
Run once each day

↓

Store result in Redis

↓

Every request uses Redis

↓

Refresh tomorrow
```

---

## 2. User Sessions

A session stores temporary information about a logged-in user.

Example:

```text
Session ID

↓

Logged in user

↓

Shopping cart

↓

Preferences
```

Instead of storing this in PostgreSQL, many websites store it in Redis.

If Redis crashes:

```text
Users are logged out.
```

Not ideal, but not catastrophic.

Because sessions are temporary, Redis is a good fit.

---

## 3. Analytics & Telemetry

Key-value stores are very good at simple arithmetic operations.

Example:

Every page visit:

```text
page_views = page_views + 1
```

Redis can perform this efficiently.

Common uses:

* Website page views.
* API request counts.
* Download counters.
* Like counters.

# Choosing a Key-Value Store

## PostgreSQL (`JSONB` and `hstore`)

PostgreSQL can also behave somewhat like a key-value store.

Older feature:

```text
hstore
```

Modern approach:

```text
JSONB
```

`JSONB` allows storing flexible key-value style data inside PostgreSQL.

Example:

```json
{
  "theme": "dark",
  "language": "en"
}
```

However, using PostgreSQL this way is generally suitable only for **small or lightweight use cases**.

Most applications use Redis to reduce load on PostgreSQL, not replace it.

---

## Memcached

Memcached is one of the oldest key-value stores.

Its philosophy is:

> Keep everything simple and extremely fast.

### Characteristics

* Everything is stored in RAM.
* Nothing is saved to disk.
* No built-in replication.
* Very lightweight.
* Extremely fast.

If memory becomes full:

```text
Old entries are removed automatically.
```

If the server restarts:

```text
Everything is lost.
```

Because of this, Memcached is mainly used for:

* Cache.
* Sessions.
* Temporary data.

---

### Data Types

Memcached only stores **strings**.

Example:

```text
"user"

"123"

"{\"name\":\"Alice\"}"
```

If you want to store objects:

Convert them into JSON strings.

Example:

```text
{
  "name": "Alice",
  "age": 30
}
```

becomes:

```text
"{\"name\":\"Alice\",\"age\":30}"
```

---

## Redis

Redis stands for:

```text
Remote Dictionary Server
```

Redis is much more feature-rich than Memcached.

### Features

* Multiple data types.
* Clustering.
* Replication.
* Persistence (can save data to disk).
* Better recovery after restart.
* Many advanced features.

Unlike Memcached:

```text
Server restarts

↓

Redis can reload saved data
```

instead of losing everything.

Redis is still commonly used for:

* Cache.
* Sessions.
* Analytics.
* Queues.
* Temporary storage.

Even though Redis can persist data, it **should not usually be your only source of important data**. Databases like PostgreSQL and MongoDB provide stronger guarantees for long-term, mission-critical storage.

---

## Valkey

Valkey is an open-source fork of Redis.

Why?

In 2024:

* Redis changed from an open-source license to a more restrictive dual-license model.
* The Linux Foundation created **Valkey** to continue the original open-source version.

The important point is:

Valkey is **API-compatible** with Redis.

That means:

* Same commands.
* Same protocol.
* Same client libraries.

Most Redis applications can switch to Valkey with little or no code changes.

Everything you learn about Redis generally applies to Valkey as well.

---

# Redis vs Memcached

| Feature                   | Redis                              | Memcached                 |
| ------------------------- | ---------------------------------- | ------------------------- |
| Stores data in memory     | Yes                                | Yes                       |
| Saves data to disk        | Yes                                | No                        |
| Multiple data types       | Yes                                | No (strings only)         |
| Replication               | Yes                                | No                        |
| Clustering                | Yes                                | Limited                   |
| Persistence after restart | Yes                                | No                        |
| Best for                  | Cache, sessions, analytics, queues | Simple cache and sessions |

---

# Key Takeaways

* A **key-value store** stores data as simple **key → value** pairs.
* It is usually a **secondary database**, not the primary one.
* It is designed for **very fast reads and writes**.
* It cannot perform complex queries like SQL databases.
* The most common use is **caching** expensive database queries.
* Other common uses include **sessions**, **analytics**, **telemetry**, and **counters**.
* **Memcached** is simple, fast, and stores everything in memory.
* **Redis** adds persistence, replication, clustering, and multiple data types.
* **Valkey** is an open-source, API-compatible alternative to Redis.

# One-Sentence Summary

**A key-value store is a fast, simple database that stores data as key-value pairs and is commonly used alongside databases like PostgreSQL or MongoDB for caching, session storage, analytics, and other temporary or frequently accessed data.**

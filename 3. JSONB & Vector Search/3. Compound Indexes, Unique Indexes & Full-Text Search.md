# Compound Indexes, Unique Indexes, and Full-Text Search — Simple Summary

PostgreSQL provides different types of indexes for different use cases.

* **Compound indexes** speed up queries that frequently use multiple columns together.
* **Unique indexes** prevent duplicate values while also improving search performance.
* **Full-text search** allows efficient searching through large amounts of text, similar to how search engines work.

Choosing the right type of index can significantly improve your application's performance.

## Key Points

* Compound indexes optimize queries using multiple columns.
* Unique indexes enforce uniqueness and improve lookups.
* Full-text search allows searching text efficiently.
* PostgreSQL supports specialized index types like **GIN** for text searching.
* For extremely advanced search features, dedicated search engines (like Elasticsearch) may be better.

---

# Compound Indexes

A **compound index** (also called a **composite index**) is an index built from **two or more columns**.

Instead of indexing each column separately, PostgreSQL stores them together in one index.

This is useful when queries frequently filter or sort using the same combination of columns.

Example query:

```sql id="z8t8fw"
SELECT *
FROM comments
WHERE board_id = 39
ORDER BY time DESC;
```

Since this query always uses both `board_id` and `time`, a compound index is a good choice.

## Creating a Compound Index

```sql id="zbw8bc"
CREATE INDEX ON comments (board_id, time);
```

Now PostgreSQL can use one index instead of combining two separate indexes.

### Benefits

* Faster queries using both columns together.
* Better performance than separate indexes for the same query pattern.

### Best Use Case

Use a compound index when the same columns are frequently used together in:

* `WHERE`
* `ORDER BY`
* `JOIN`

---

# Unique Indexes

A **unique index** guarantees that every value in a column is unique.

Example:

Usernames should never be duplicated.

Instead of checking duplicates in application code, PostgreSQL enforces the rule automatically.

## Creating a Unique Index

```sql id="eg8a2g"
CREATE UNIQUE INDEX username_idx
ON users (username);
```

### Explanation

* `CREATE UNIQUE INDEX` → Creates an index that also enforces uniqueness.
* `username_idx` → Name of the index.
* `ON users` → Creates the index on the `users` table.
* `(username)` → The indexed column.

---

# What Happens with Duplicate Data?

Attempting to insert an existing username:

```sql id="tznb90"
INSERT INTO users
(username, email, full_name, created_on)
VALUES
(
  'aaizikovj',
  'lol@example.com',
  'Brian Holt',
  NOW()
);
```

Result:

```text id="znpvh7"
ERROR:
duplicate key value violates unique constraint
```

The insert fails because the username already exists.

As a bonus, searching by username is also very fast because the column is indexed.

# Full-Text Search

Sometimes you want to search large amounts of text.

Example:

Find comments containing:

* love
* dog
* programming

A normal `LIKE` search isn't ideal because it:

* Is slower.
* Doesn't understand language.
* Doesn't ignore common words.
* Doesn't support advanced searching.

PostgreSQL includes **built-in full-text search**.

---

# Basic Full-Text Search

```sql id="d6bjz2"
SELECT
  comment_id,
  LEFT(comment, 50)
FROM comments
WHERE to_tsvector('english', comment)
@@
to_tsquery('english', 'love');
```

This searches for comments containing the word **love**.

---

# How It Works

## `to_tsvector()`

Converts text into searchable words called **lexemes**.

Example:

```text id="eywobn"
"I loved programming."
```

Becomes something like:

```text id="rpk2wi"
love
program
```

Notice that:

* Different word forms are normalized.
* Common words (such as "the", "a", "and") are ignored.

---

## `to_tsquery()`

Converts a search phrase into a searchable query.

Example:

```sql id="02kmdg"
to_tsquery('english', 'love')
```

Creates a search query for the word **love**.

---

## `@@`

The match operator.

Example:

```sql id="o8zkjj"
to_tsvector(...)
@@
to_tsquery(...)
```

Means:

> Does this searchable text match this search query?

---

# Why Is It Slow Initially?

Without an index:

```text id="g5kwm7"
Every row
       │
       ▼
Create tsvector
       │
       ▼
Compare with search term
```

The database performs this work for every row.

---

# Speeding It Up with a GIN Index

Create a full-text search index:

```sql id="k3nkz2"
CREATE INDEX comments_search_idx
ON comments
USING GIN
(
  to_tsvector('english', comment)
);
```

Now PostgreSQL stores searchable text in a specialized index.

Searches become much faster.

---

# What is a GIN Index?

**GIN** stands for **Generalized Inverted Index**.

It is designed for searching values inside collections such as:

* Text
* JSONB
* Arrays

Instead of indexing entire rows, it indexes individual searchable terms.

Example:

```text id="jpxi57"
love
 ├── Comment 4
 ├── Comment 27
 └── Comment 105
```

Searching becomes extremely efficient.

---

# Searching with AND

Find comments containing **love** and **dog**.

```sql id="wry4mz"
SELECT
  comment_id,
  LEFT(comment, 50)
FROM comments
WHERE
to_tsvector('english', comment)
@@
to_tsquery('english', 'love & dog');
```

The `&` operator means **AND**.

Both words must be present.

---

# Searching with OR

Find comments containing **love** or **hate**.

```sql id="9nrhpl"
SELECT
  comment_id,
  LEFT(comment, 50)
FROM comments
WHERE
to_tsvector('english', comment)
@@
to_tsquery('english', 'love | hate');
```

The `|` operator means **OR**.

Either word may appear.

---

# PostgreSQL vs Dedicated Search Engines

PostgreSQL's built-in full-text search is powerful enough for many applications.

Good for:

* Blogs
* Forums
* Documentation
* Small to medium websites

For extremely advanced search features such as:

* Typo correction
* Ranking and relevance tuning
* Faceted filtering
* Massive search workloads

Dedicated search engines are usually better.

Examples:

* Elasticsearch
* OpenSearch

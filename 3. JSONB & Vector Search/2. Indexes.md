# Database Indexes — Simple Summary

Databases are already highly optimized and can handle many queries efficiently without extra configuration. However, as applications grow and tables contain millions of rows, some queries become slow because the database has to examine too much data.

An **index** is a special data structure that helps the database find rows much faster, similar to how a book's index helps you quickly locate a topic instead of reading every page.

Indexes improve **read (SELECT)** performance but come with trade-offs: they consume extra storage and slightly slow down **INSERT**, **UPDATE**, and **DELETE** operations because the index must also be updated.

## Key Points

* Databases are already highly optimized.
* Some queries become slow as data grows.
* Indexes speed up searches.
* Indexes are separate data structures maintained by the database.
* Faster reads come at the cost of slower writes and additional disk space.
* Indexes can also enforce uniqueness (such as unique email addresses).

---

# Why Are Indexes Needed?

Imagine looking for one person's name in a phone book.

Without an index:

```text
Read page 1
Read page 2
Read page 3
...
Read every page
```

With an index:

```text
Open index
Find the page number
Go directly to that page
```

Databases work the same way.

Without an index, PostgreSQL may have to examine every row.

With an index, it can jump directly to the matching rows.

---

# Before Using an Index

Suppose you want the latest 40 comments from a specific message board.

```sql
SELECT
  comment_id,
  user_id,
  time,
  LEFT(comment, 20)
FROM comments
WHERE board_id = 39
ORDER BY time DESC
LIMIT 40;
```

This query is simple, but PostgreSQL may need to scan the entire `comments` table if no index exists.

On a table with millions of rows, this can become very slow.

---

# Understanding `EXPLAIN`

`EXPLAIN` shows **how PostgreSQL plans to execute a query**.

It doesn't run the query—it simply displays the execution plan.

Example:

```sql
EXPLAIN
SELECT
  comment_id,
  user_id,
  time,
  LEFT(comment, 20)
FROM comments
WHERE board_id = 39
ORDER BY time DESC
LIMIT 40;
```

The output may include:

```text
Seq Scan on comments
```

---

# What is a Sequential Scan?

A **Sequential Scan (Seq Scan)** means PostgreSQL reads every row in the table until it finds the matching ones.

Example:

```text
Row 1
Row 2
Row 3
...
Row 1,000,000
```

This is essentially a **linear search**.

Time complexity:

```text
O(n)
```

As the table grows, performance becomes worse.

---

# Creating an Index

To speed up searches on `board_id`:

```sql
CREATE INDEX ON comments (board_id);
```

This tells PostgreSQL to build an index for the `board_id` column.

The database creates a separate structure that allows fast lookups.

---

# After Creating the Index

Run the same `EXPLAIN` again:

```sql
EXPLAIN
SELECT
  comment_id,
  user_id,
  time,
  LEFT(comment, 20)
FROM comments
WHERE board_id = 39
ORDER BY time DESC
LIMIT 40;
```

Now PostgreSQL may show:

```text
Bitmap Heap Scan on comments
```

Instead of scanning the whole table, PostgreSQL uses the index to locate the matching rows much more efficiently.

---

# Bitmap Heap Scan

A **Bitmap Heap Scan** is an optimized access method.

Instead of checking every row:

1. PostgreSQL looks in the index.
2. Finds the matching row locations.
3. Reads only those rows from the table.

Example:

```text
Index
 │
 ├── Row 152
 ├── Row 918
 ├── Row 2045
 │
 ▼
Read only those rows
```

This is much faster than reading the entire table.

---

# Viewing Existing Indexes

To list all indexes on a table:

```sql
SELECT
  indexname,
  indexdef
FROM pg_indexes
WHERE tablename = 'comments';
```

This returns information such as:

```text
indexname                 indexdef
------------------------  ---------------------------------------
comments_pkey             CREATE UNIQUE INDEX ...
comments_board_id_idx     CREATE INDEX ...
```

This helps you see which indexes already exist.

---

# Benefits of Indexes

Advantages:

* Much faster `SELECT` queries.
* Faster searching.
* Faster filtering (`WHERE`).
* Faster sorting in many cases.
* Can enforce unique values.

---

# Trade-offs of Indexes

Disadvantages:

* Consume additional disk space.
* Slow down `INSERT`.
* Slow down `UPDATE`.
* Slow down `DELETE`.

This happens because every data change must also update the index.

---

# When Should You Create an Index?

Indexes are useful when:

* A column is frequently searched.
* A column is often used in `WHERE`.
* A column is frequently used in `JOIN`.
* A column is commonly used in `ORDER BY`.
* The table contains a large amount of data.

Indexes may not be worthwhile when:

* The table is very small.
* The column changes constantly.
* The column is rarely queried.

---

# Common Query Flow

Without an index:

```text
Query
   │
   ▼
Read every row
   │
   ▼
Find matching rows
```

With an index:

```text
Query
   │
   ▼
Use index
   │
   ▼
Jump directly to matching rows
```

---

# One-Sentence Summary

**An index is a separate data structure that helps PostgreSQL find data much faster by avoiding full table scans, improving query performance at the cost of additional storage and slightly slower insert, update, and delete operations.**

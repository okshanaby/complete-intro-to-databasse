# Vector Search and RAG — Simple Summary

**Vector search** is a technique for finding information based on **meaning** instead of exact keyword matches.

It is most commonly used with **RAG (Retrieval-Augmented Generation)**, where relevant information is retrieved from a database and given to a Large Language Model (LLM) before it generates an answer.

Instead of relying only on what the AI already knows, RAG allows the AI to answer using your own data.

## Key Points

* Vector search finds information by meaning, not exact words.
* RAG combines database retrieval with AI generation.
* Text is converted into vectors (arrays of numbers).
* Similar meanings produce similar vectors.
* Retrieved information is provided as context to the LLM.
* PostgreSQL can support vector search using extensions like **pgvector**.

---

# What is RAG?

**RAG** stands for **Retrieval-Augmented Generation**.

It is a technique where an AI first retrieves relevant information from a database and then uses that information to generate a better response.

Instead of answering immediately:

```text id="m4sk9a"
User Question
      │
      ▼
LLM
      │
      ▼
Answer
```

A RAG system works like this:

```text id="g4b2q7"
User Question
      │
      ▼
Vector Search
      │
Retrieve Relevant Data
      │
      ▼
LLM
      │
      ▼
Answer Based on Retrieved Context
```

The retrieved information becomes additional context for the AI.

# Example: Outdoor Equipment Store

Suppose a customer asks:

```text id="3hyqyo"
Which gloves are best for warm spring skiing?
```

Your database contains hundreds of glove descriptions.

Instead of sending only the question to the AI:

```text id="dlkmku"
Question
   │
   ▼
LLM
```

The system first searches the database.

```text id="6n7z2u"
Question
   │
   ▼
Vector Search
   │
Find Similar Products
   │
   ▼
LLM
```

The AI now has product information before generating its recommendation.

This leads to more accurate answers based on products you actually sell.


# What is a Vector?

A **vector** is simply an array of numbers.

Example:

```text id="ynjlwm"
[0.42, -1.17, 0.91, 0.08, ...]
```

Humans cannot interpret these numbers directly.

Instead, they represent the meaning of the text.

---

# How Text Becomes a Vector

Text is sent to an **embedding model**.

Example:

```text id="nq3n9n"
"I love skiing."
      │
      ▼
Embedding Model
      │
      ▼
[0.21, -0.43, 1.82, ...]
```

The embedding model converts text into numerical vectors.

These vectors are then stored in the database.

---

# What is an Embedding Model?

An **embedding model** is a specialized AI model whose job is to convert text into vectors.

It does **not** generate answers.

Its only purpose is to produce numerical representations of meaning.

Example:

```text id="nl76iq"
Input Text
     │
     ▼
Embedding Model
     │
     ▼
Vector
```

---

# Why Are Vectors Useful?

Traditional keyword search only matches exact words.

Example:

```text id="l3shtn"
love skiing
```

won't necessarily match

```text id="qqnqbl"
enjoy snow sports
```

because they use different words.

Vector search understands meaning instead.

Example:

```text id="jlwm23"
"Pushing to production on Friday is stressful."
```

and

```text id="jlwm24"
"Weekend deployments make me anxious."
```

These sentences share very few keywords, but they express a similar idea.

Their vectors are close together, so vector search considers them related.

---

# Similarity Search

Instead of searching for exact words, vector search finds vectors that are closest together.

Conceptually:

```text id="jlwm25"
Question Vector
      │
      ▼
Compare Distance
      │
      ▼
Closest Vectors
```

Closer vectors represent more similar meanings.

---

# Vector Search vs Keyword Search

| Keyword Search             | Vector Search                    |
| -------------------------- | -------------------------------- |
| Matches exact words        | Matches meaning                  |
| May miss synonyms          | Understands similar concepts     |
| Doesn't understand context | Captures semantic similarity     |
| Best for exact matching    | Best for natural language search |

# Vector Search and PostgreSQL

Previously, developers often needed specialized vector databases.

Examples:

* Pinecone

Now PostgreSQL can perform vector search using extensions such as:

* **pgvector**

This allows vector data to live alongside your relational data in the same PostgreSQL database.

Benefits include:

* One database instead of multiple systems.
* Easier maintenance.
* Simpler architecture.
* Less data synchronization.

---

# When Should You Use Vector Search?

Use vector search when:

* Building RAG applications.
* AI chatbots.
* Customer support assistants.
* Semantic document search.
* Recommendation systems.
* Knowledge base search.

Avoid vector search when:

* Exact keyword matching is required.
* Traditional full-text search is sufficient.
* You need highly optimized search features like typo correction or faceted search.

---

# Vector Search vs Full-Text Search

| Full-Text Search             | Vector Search                 |
| ---------------------------- | ----------------------------- |
| Matches words                | Matches meaning               |
| Very fast for keyword search | Excellent for semantic search |
| Great for documents          | Great for AI applications     |
| Often uses GIN indexes       | Uses vector indexes           |

For very advanced keyword search features, dedicated search engines such as **Elasticsearch** are still commonly used.

# One-Sentence Summary

**Vector search converts text into numerical vectors that represent meaning, allowing RAG systems to retrieve semantically similar information from a database before sending it to an LLM, resulting in more accurate and context-aware AI responses.**

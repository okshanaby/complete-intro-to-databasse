# Node.js App with PostgreSQL — Simple Summary

This example demonstrates how to connect a **Node.js** application to a **PostgreSQL** database using the **pg** library and **Express**.

The application has:

* A simple HTML frontend
* An Express backend
* A PostgreSQL database

The frontend sends a board ID to the backend, the backend queries PostgreSQL, and the results are returned as JSON.

A major lesson in this section is **using parameterized queries** to prevent **SQL Injection** attacks.

## Key Points

* Express creates the web server.
* `pg` connects Node.js to PostgreSQL.
* A connection pool efficiently manages database connections.
* Parameterized queries (`$1`) protect against SQL injection.
* Never place user input directly inside SQL strings.

# Backend (`server.js`)

```js
import express from "express";
import { Pool } from "pg";
const pool = new Pool({
  connectionString: "postgresql://postgres:mysecretpassword@localhost:5432/postgres",
});

async function init() {
  const app = express();

  app.get("/get", async (req, res) => {
    const client = await pool.connect();
    const [commentsRes, boardRes] = await Promise.all([
      client.query(
        // `SELECT * FROM comments NATURAL LEFT JOIN rich_content WHERE board_id = ${req.query.search}`
        "SELECT * FROM comments NATURAL LEFT JOIN rich_content WHERE board_id = $1",
        [req.query.search],
      ),
      client.query("SELECT * FROM boards WHERE board_id = $1", [
        req.query.search,
      ]),
    ]);
    res
      .json({
        status: "ok",
        board: boardRes.rows[0] || {},
        posts: commentsRes.rows,
      })
      .end();
    client.release(); // don't use client.end()
  });

  const PORT = process.env.PORT || 3000;
  app.use(express.static("./static"));
  app.listen(PORT);

  console.log(`running on http://localhost:${PORT}`);
}
init();
```

The server:

1. Starts Express.
2. Connects to PostgreSQL.
3. Receives requests.
4. Executes SQL queries.
5. Returns JSON.

Connection pool:

```javascript
const pool = new Pool({
  connectionString:
    "postgresql://db_user:db_password@localhost:5432/db_name",
});
```

Instead of opening a new connection every request, the pool reuses existing connections.

This is much faster.

# Connection Pool

A **connection pool** is a collection of reusable database connections.

Without a pool:

```text
Request
   │
Open connection
   │
Run query
   │
Close connection
```

This happens every request.

With a pool:

```text
Request
   │
Reuse connection
   │
Run query
   │
Return connection to pool
```

Benefits:

* Faster
* Less overhead
* Fewer database connections
* Better scalability

# Executing Multiple Queries

The example executes two queries simultaneously.

```javascript
const [commentsRes, boardRes] =
await Promise.all([
  client.query(...),
  client.query(...)
]);
```

Instead of waiting:

```text
Query 1
↓

Query 2
```

They run together.

```text
Query 1
      ↘
        Finish
      ↗
Query 2
```

This improves performance.

# Releasing the Connection

After finishing:

```javascript
client.release();
```

This returns the connection to the pool.

Do **not** use:

```javascript
client.end();
```

because that permanently closes the connection instead of reusing it.

# Why Avoid `SELECT *`?

The example uses:

```sql
SELECT *
FROM comments;
```

This is acceptable for learning, but in production you should request only the columns you need.

Instead of:

```sql
SELECT *
FROM comments;
```

Prefer:

```sql
SELECT
comment_id,
user_id,
comment
FROM comments;
```

Benefits:

* Less data transferred.
* Faster queries.
* Better performance.

# SQL Injection

The most important lesson is avoiding **SQL Injection**.

SQL Injection happens when user input becomes part of an SQL command.

Unsafe example:

```javascript
client.query(
`SELECT *
FROM comments
WHERE board_id = ${req.query.search}`
);
```

If the user enters:

```text
39
```

The SQL becomes:

```sql
SELECT *
FROM comments
WHERE board_id = 39;
```

This is fine.

---

# The Problem

Suppose someone enters:

```text
1; SELECT * FROM users; --
```

The SQL becomes:

```sql
SELECT *
FROM comments
WHERE board_id = 1;

SELECT *
FROM users;
```

The attacker now retrieves user data.

Even worse:

```text
1; DROP TABLE users; --
```

Produces:

```sql
SELECT *
FROM comments
WHERE board_id = 1;

DROP TABLE users;
```

The users table could be deleted.

---

# Safe Parameterized Queries

Instead of inserting user input directly into SQL:

```javascript
client.query(
"SELECT * FROM comments WHERE board_id = $1",
[req.query.search]
);
```

Here:

* `$1` is a placeholder.
* PostgreSQL safely inserts the value.

Even if the user enters:

```text
1; DROP TABLE users;
```

PostgreSQL treats it as plain text.

The SQL command itself never changes.

This completely prevents SQL Injection.

---

# Why Parameterized Queries Are Safe

Unsafe:

```text
User Input
      │
      ▼
SQL String
```

Safe:

```text
User Input
      │
      ▼
Parameter
      │
      ▼
Database
```

The database knows the input is **data**, not SQL code.

# Best Practices Learned

* Use a connection pool.
* Release connections after use.
* Run independent queries with `Promise.all()`.
* Select only required columns.
* Always use parameterized queries.
* Never trust user input.

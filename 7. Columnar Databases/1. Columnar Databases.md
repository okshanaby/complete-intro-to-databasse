# Columnar Databases — Simple Summary

**Columnar databases** are databases designed for **analytics and reporting**, not for running everyday web applications.

Unlike databases such as PostgreSQL or MySQL that store data **row by row**, columnar databases store data **column by column**. This makes them extremely fast when analyzing large datasets.

Popular columnar databases include:

* DuckDB
* ClickHouse
* Snowflake
* Databricks Lakehouse

They are commonly used for:

* Business intelligence (BI)
* Dashboards
* Reporting
* Data science
* Analytics
* Machine learning data preparation

---

# Key Points

* Optimized for **reading large amounts of data**
* Best for **aggregations** (SUM, COUNT, AVG, etc.)
* Not intended to replace transactional databases like PostgreSQL
* Usually works alongside an OLTP database
* Reads only the columns needed for a query
* Excellent for analyzing millions or billions of records

---

# Row-Based vs Column-Based Storage

Traditional databases (PostgreSQL, MySQL, etc.) store data **row by row**.

Example table:

| ID | Name  | Age | Email                                     |
| -- | ----- | --- | ----------------------------------------- |
| 1  | Alice | 25  | [alice@email.com](mailto:alice@email.com) |
| 2  | Bob   | 31  | [bob@email.com](mailto:bob@email.com)     |
| 3  | John  | 40  | [john@email.com](mailto:john@email.com)   |

---

## Row Database Storage

A row database stores records like this:

```text
Row 1:
1 | Alice | 25 | alice@email.com

Row 2:
2 | Bob | 31 | bob@email.com

Row 3:
3 | John | 40 | john@email.com
```

Each row contains the complete record.

This is perfect for application requests like:

```text
Give me user #2
```

The database reads one complete row.

---

## Column Database Storage

A columnar database stores data like this:

```text
ID
1
2
3

Name
Alice
Bob
John

Age
25
31
40

Email
alice@email.com
bob@email.com
john@email.com
```

Each column is stored separately.

---

# Why Column Storage Is Faster

Imagine a table with **10 million users**.

You want to calculate:

```text
Average age of all users
```

A **row database** reads:

```text
ID
Name
Age
Email
```

for every user, even though only the **Age** column is needed.

Visualization:

```text
Read Row 1
 ├── ID
 ├── Name
 ├── Age
 └── Email

Read Row 2
 ├── ID
 ├── Name
 ├── Age
 └── Email

...
```

Much of the data read is unnecessary.

---

A **column database** reads only:

```text
Age
25
31
40
...
```

Visualization:

```text
Read Age Column

25
31
40
28
35
22
...
```

Only the required column is loaded into memory.

Result:

* Less disk I/O
* Less memory usage
* Faster analytical queries

---

# The Tradeoff

Columnar databases are slower when retrieving a complete record.

Suppose you want:

```text
User #2
```

A row database reads:

```text
Row 2

2
Bob
31
bob@email.com
```

Done.

---

A column database must read:

```text
ID column

↓

Name column

↓

Age column

↓

Email column
```

Then reconstruct:

```text
2
Bob
31
bob@email.com
```

This extra work makes them inefficient for transactional applications.

---

# Why You Don't Build Apps on Columnar Databases

Imagine a social media website.

Every page load needs:

* User profile
* Username
* Password
* Email
* Settings
* Friends

These are complete records.

A row database retrieves them quickly.

A column database must gather data from many separate columns every time.

For this reason:

* PostgreSQL → good for applications
* DuckDB → good for analytics

# OLTP vs OLAP

These are two important database categories.

| OLTP                          | OLAP                         |
| ----------------------------- | ---------------------------- |
| Online Transaction Processing | Online Analytical Processing |
| Runs applications             | Performs analytics           |
| Frequent small reads/writes   | Large analytical queries     |
| Optimized for transactions    | Optimized for reporting      |
| Row-oriented                  | Usually column-oriented      |

---

## OLTP Examples

General-purpose databases:

* PostgreSQL
* MySQL
* Oracle
* MongoDB
* SQLite

Typical queries:

```text
Get user #42

Insert order

Update inventory

Delete session
```

These happen constantly in applications.

---

## OLAP Examples

Analytical databases:

* DuckDB
* ClickHouse
* Snowflake
* Databricks Lakehouse

Typical queries:

```text
Average sales

Revenue by month

Most popular products

Customer behavior

Regional trends
```

These process huge datasets efficiently.

---

# Typical Architecture

Applications often use both database types.

```text
            Users
              │
              ▼
      Web Application
              │
              ▼
      PostgreSQL (OLTP)
              │
     Export application data
              │
              ▼
      DuckDB / ClickHouse (OLAP)
              │
              ▼
 Dashboards, Reports, Analytics
```

The application serves users from PostgreSQL, while analytics are performed in the OLAP database.

# ETL (Extract, Transform, Load)

**ETL** is a process for moving data into an analytical database.

It has three steps:

1. Extract
2. Transform
3. Load

Visualization:

```text
PostgreSQL
      │
      ▼
Extract Data
      │
      ▼
Transform Data
      │
      ▼
DuckDB
```

---

## Step 1 — Extract

Copy data from the source.

Example:

```text
PostgreSQL

↓

Export CSV
```

---

## Step 2 — Transform

Prepare the data.

Examples:

* Remove duplicates
* Rename columns
* Convert dates
* Merge tables
* Clean invalid values

---

## Step 3 — Load

Import the cleaned data into the analytical database.

Now reports can be generated efficiently.

---

# ELT (Extract, Load, Transform)

ELT changes the order.

Instead of transforming first:

```text
Extract

↓

Load

↓

Transform
```

The raw data is loaded immediately.

Transformations happen later when querying.

---

## Why ELT Is Popular

Suppose your application receives new orders every second.

With ETL:

```text
Extract every hour

↓

Transform

↓

Load
```

Reports may be **up to one hour old**.

---

With ELT:

```text
Extract

↓

Load immediately

↓

Transform during queries
```

Reports stay much closer to real-time because data is available sooner.

---

# DuckDB Makes This Easy

One advantage of DuckDB is that it works directly with files.

It can read formats like:

* CSV
* JSON
* Parquet
* Other supported storage files

Example workflow:

```text
PostgreSQL

↓

Export CSV

↓

DuckDB reads CSV directly

↓

Run analytics
```

This makes it simple to start analyzing data without setting up a complex data warehouse.

---

# Reverse-ETL

Normally data flows:

```text
OLTP

↓

OLAP
```

Reverse-ETL sends processed analytical results back into the application's database.

Visualization:

```text
PostgreSQL
      │
      ▼
DuckDB
      │
Analyze Data
      │
Generate Insights
      │
      ▼
Back to PostgreSQL
```

---

## Example

An online store stores purchases in PostgreSQL.

The sales data is copied into DuckDB.

DuckDB discovers:

```text
Customers who buy Laptop

↓

Often buy Mouse
```

This recommendation is useful for the website.

The recommendation is written back into PostgreSQL.

Now the application can display:

```text
Customers also bought:

• Mouse
• Keyboard
• Laptop Bag
```

This process is called **Reverse-ETL** because processed analytical data is moved from the OLAP database back into the OLTP database for use by the application.

# When to Use Each Database

| Use Case                       | Best Choice       |
| ------------------------------ | ----------------- |
| User accounts                  | PostgreSQL (OLTP) |
| Orders                         | PostgreSQL (OLTP) |
| Login system                   | PostgreSQL (OLTP) |
| Analytics                      | DuckDB (OLAP)     |
| Dashboards                     | DuckDB (OLAP)     |
| Business reports               | DuckDB (OLAP)     |
| Machine learning data analysis | DuckDB (OLAP)     |

---

# One-Sentence Summary

**Columnar databases store data by columns instead of rows, making them extremely fast for analytical queries and reporting, while traditional row-based databases remain the best choice for transactional web applications.**

# Choosing the Right Database — Simple Summary

Choosing a database isn't about picking the newest or most popular technology. It's about choosing the one that best fits your application's data, workload, team, and operational needs.

There is **no single "best" database**. Different databases solve different problems.

---

# Key Points

* Choose a database based on your data model.
* Consider how your application reads and writes data.
* Factor in your team's experience.
* Check the quality of drivers/SDKs for your programming language.
* Consider operational and maintenance costs.
* Prefer stable, well-tested technologies for core infrastructure.
* Use caching only when it's truly needed.

---

# 1. Choose Based on the Data Paradigm

The first question should always be:

> **What kind of data am I storing?**

Different databases are optimized for different types of data.

| Data Type                        | Best Choice                    | Why                                               |
| -------------------------------- | ------------------------------ | ------------------------------------------------- |
| Highly related data              | Relational (PostgreSQL, MySQL) | Efficient joins and relationships                 |
| Flexible or changing data        | Document (MongoDB)             | Different documents can have different structures |
| Relationship networks            | Graph (Neo4j)                  | Optimized for connected data                      |
| Simple key lookups               | Key-Value (Redis)              | Extremely fast retrieval by key                   |
| Large-scale analytics            | Columnar (DuckDB, ClickHouse)  | Fast aggregations and reporting                   |
| Data pipelines and event streams | Apache Kafka                   | Processing and distributing streaming data        |

# 2. Read-Heavy vs Write-Heavy Workloads

Think about how your application uses its data.

Questions to ask:

* Is the application reading data constantly?
* Is it writing data constantly?
* Which operations happen most often?

Optimize for the busiest parts of your application.

---

## Read-Heavy Example

```text id="rqzpvk"
Users

↓

Read Product Catalog

↓

Thousands of times
```

If reads dominate, optimize read performance.

---

## Write-Heavy Example

```text id="mjlwmh"
Sensors

↓

Thousands of Writes

↓

Every Second
```

Here, write performance becomes the priority.

---

## Practical Advice

Sometimes a database isn't perfect for every workload, but it's still the right choice overall.

Example:

* Most data fits MongoDB well.
* A few features need joins.

Since joins are infrequent, MongoDB may still be the better overall solution.

The important principle is:

> Optimize for what happens most frequently.

# 3. Familiarity Matters

Technology isn't chosen only by technical features.

Your team's experience is also important.

Example:

Team knows:

```text id="74nglx"
MySQL
```

Nobody knows:

```text id="cvjlwm"
PostgreSQL
```

Even if PostgreSQL offers advantages, choosing MySQL may reduce:

* Development time
* Mistakes
* Downtime
* Training costs

Experience has real value.

# 4. Quality of Drivers (SDKs)

Every database needs a client library (driver/SDK) so your application can communicate with it.

Before choosing a database, verify that your programming language has:

* Good documentation
* Active maintenance
* Stable releases
* Community support

Poor SDKs can make even an excellent database frustrating to use.

Things to manage include:

* Backups
* Monitoring
* Scaling
* Security
* Networking
* Software updates
* High availability
* Disaster recovery

These operational tasks consume both time and money.

# 6. Be Boring

One of the strongest recommendations in the notes is:

> **Choose boring technology.**

"Boring" means:

* Proven
* Stable
* Well documented
* Widely used
* Reliable

Not:

* Brand new
* Experimental
* Trending on social media

## Why?

Stable technologies have:

* Better documentation
* More tutorials
* Larger communities
* More bug fixes
* Better AI assistance
* Fewer unexpected problems

# 7. Caching

Caching stores data temporarily to avoid repeating expensive operations.

Example:

```text id="4jlwm5"
Application

↓

Cache

↓

Database
```

If data exists in the cache:

```text id="jlwm9t"
Return Cached Data
```

Otherwise:

```text id="jlwm7r"
Query Database

↓

Store Result in Cache

↓

Return Result
```

---

## Why Be Careful with Caching?

Caching is powerful, but it introduces complexity.

Questions you must answer include:

* What should be cached?
* How long should it stay cached?
* When should cached data expire?
* When should it be refreshed?
* How do you keep cached data consistent?

These are difficult problems.

---

## Caching Can Hide Real Problems

Sometimes developers add a cache because the database is slow.

But the real issue may be:

* Poor SQL queries
* Missing indexes
* Bad schema design
* Incorrect configuration

Instead of fixing the underlying issue, caching only masks it.

A better approach is:

```text id="jlwm1s"
Slow Database

↓

Find Root Cause

↓

Optimize Queries

↓

Improve Database

↓

Add Cache Only If Still Needed
```

---

## Cache Invalidation

One of the hardest problems in software engineering is knowing when cached data becomes outdated.

Example:

```text id="jlwmxm"
Database

↓

Price = $50
```

Cache:

```text id="jlwm2f"
Price = $50
```

Database changes:

```text id="jlwm5d"
Price = $60
```

If the cache isn't updated:

```text id="jlwm8u"
Application

↓

Still Shows $50
```

This is called **stale cache**.

---

## Multiple Cache Layers

Some applications cache at several levels.

Example:

```text id="jlwmkv"
Browser Cache

↓

API Cache

↓

Redis Cache

↓

Database
```

When incorrect data appears, debugging becomes much harder because any cache layer could be serving outdated data.

---

## Thundering Herd

Another challenge is the **thundering herd problem**.

Example:

```text id="jlwmow"
Cache Expires

↓

Thousands of Users Request Data

↓

All Hit Database

↓

Database Overloaded
```

This can happen if many requests attempt to regenerate the same cache simultaneously.

---

## When Should You Cache?

Cache only when:

* Performance measurements show it's needed.
* Database optimization alone isn't sufficient.
* The complexity introduced by caching is justified.

Remember:

> Databases are already highly optimized. Good schema design, proper indexing, and efficient queries often eliminate the need for caching.

# Database Selection Checklist

Before choosing a database, ask yourself:

* What kind of data am I storing?
* How will the application access the data?
* Is the workload read-heavy or write-heavy?
* Does my team already know this database?
* Are the client libraries well supported?
* Can we operate and maintain it effectively?
* Is this a stable, mature technology?
* Do we actually need caching?

---

# One-Sentence Summary

**Choosing the right database means matching the database to your application's data model, workload, team expertise, and operational needs, while favoring stable technologies and using caching only after optimizing the underlying system.**

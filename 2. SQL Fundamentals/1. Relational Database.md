# Relational Database — Simple Summary

A **relational database** stores data in **tables** made up of **rows** and **columns**, much like a spreadsheet (such as Excel or Google Sheets).

* Each **row** represents one record (or entry).
* Each **column** represents one field (a specific type of information).

Relational databases use a **fixed schema**, meaning the structure of the data must be defined before data can be stored.

Their biggest strength is that **multiple tables can be related to each other**, allowing data to be connected and queried efficiently using **JOINs**.

## Key Points

* Data is stored in tables.
* Rows represent records.
* Columns represent fields.
* Uses a predefined (fixed) schema.
* Tables can be connected through relationships.
* Excellent for structured data.

## Spreadsheet Analogy

Think of a relational database like a spreadsheet.

```text
Customers Table

+----+--------+----------+-------+
| ID | Name   | City     | State |
+----+--------+----------+-------+
| 1  | Brian  | Seattle  | WA    |
| 2  | Alice  | New York | NY    |
| 3  | John   | Austin   | TX    |
+----+--------+----------+-------+
```

* Each **row** is one customer.
* Each **column** stores one type of information.

## Fixed Schema

Unlike document databases (such as MongoDB), relational databases require a predefined structure.

Example schema:

```text
Customers

ID
Name
City
State
```

Valid record:

```text
ID: 1
Name: Brian
City: Seattle
State: WA
```

Invalid attempt:

```text
ID: 2
Name: Alice
City: New York
State: NY
Comment: VIP Customer
```

The `Comment` field is rejected because it isn't part of the schema.

### Advantages

* Keeps data organized.
* Prevents unexpected data.
* Makes applications more consistent.

### Disadvantages

* Less flexible.
* Changing the schema usually requires updating the database structure.

# Relationships Between Tables

The word **relational** comes from the ability to relate tables together.

Instead of storing everything in one huge table, data can be split into multiple related tables.

Example:

```text
Customers

+----+--------+
| ID | Name   |
+----+--------+
| 1  | Brian  |
| 2  | Alice  |
+----+--------+
```

```text
Orders

+----+-------------+--------+
| ID | CustomerID  | Total  |
+----+-------------+--------+
|101 |      1      | 50.00  |
|102 |      1      |120.00  |
|103 |      2      | 75.00  |
+----+-------------+--------+
```

Here:

* `CustomerID` connects each order to a customer.
* One customer can have many orders.

This relationship allows databases to combine information when needed.

## JOINs

A **JOIN** combines related data from multiple tables.

Instead of duplicating customer information in every order, the database links the tables together.

Example result after joining:

```text
+--------+----------+--------+
| Name   | Order ID | Total  |
+--------+----------+--------+
| Brian  |   101    | 50.00  |
| Brian  |   102    |120.00  |
| Alice  |   103    | 75.00  |
+--------+----------+--------+
```

This is one of the biggest advantages of relational databases.

# Relational Database vs Document Database

| Relational Database      | Document Database                                  |
| ------------------------ | -------------------------------------------------- |
| Stores data in tables    | Stores data as documents                           |
| Fixed schema             | Flexible schema                                    |
| Excellent at JOINs       | JOINs are generally discouraged across collections |
| Best for structured data | Best for rapidly changing or flexible data         |

# What is an RDBMS?

**RDBMS** stands for **Relational Database Management System**.

An RDBMS is the software that creates, manages, and interacts with relational databases.

Examples include:

* PostgreSQL
* MySQL
* SQLite
* Microsoft SQL Server
* Oracle Database

An RDBMS handles tasks such as:

* Storing data
* Managing tables
* Enforcing schemas
* Processing SQL queries
* Maintaining relationships between tables

## One-Sentence Summary

**A relational database stores structured data in tables with rows and columns, enforces a fixed schema, and allows multiple tables to be connected through relationships and JOINs, all managed by an RDBMS.**

# JOIN in PostgreSQL — Simple Summary

A **JOIN** is used when you need to combine data from multiple tables into a single result.

In real applications, data is usually split across tables (normalization). JOIN lets you reconnect that data when querying.

---

# Problem Example (Before JOIN)

```sql id="j1"
SELECT comment_id, user_id, LEFT(comment, 20) AS preview
FROM comments
WHERE board_id = 39;
```

---

## What this query does

Returns comments for a specific board with a short preview.

---

## Breakdown

### `LEFT(comment, 20)`

```text id="j2"
LEFT(text, n)
```

* Takes the first `n` characters from a string
* Here: first 20 characters of the comment

---

### `AS preview`

```text id="j3"
LEFT(comment, 20) AS preview
```

* Renames the output column
* Makes result easier to read

Without it, PostgreSQL would name it something like:

```text id="j4"
left
```

which is not meaningful.

---

## Problem with this result

You only get:

* `user_id` (not readable)
* No username

So users see:

```text id="j5"
user_id = 858
```

But they want:

```text id="j6"
username = "brian"
```

---

# Solution: INNER JOIN

```sql id="j7"
SELECT
  comment_id,
  comments.user_id,
  users.username,
  time,
  LEFT(comment, 20) AS preview
FROM
  comments
INNER JOIN
  users
ON
  comments.user_id = users.user_id
WHERE
  board_id = 39;
```

---

## What this query does

Combines data from:

* `comments` table
* `users` table

So each comment includes the username of the person who wrote it.

---

# INNER JOIN Breakdown

## `INNER JOIN`

```text id="j8"
INNER JOIN users
```

### What it means

* Match rows from both tables
* Only return rows where a match exists in BOTH tables

---

## `ON` clause

```sql id="j9"
ON comments.user_id = users.user_id
```

### What it does

Defines how the tables connect:

* `comments.user_id` → foreign key
* `users.user_id` → primary key

So:

> “Match comments to users where IDs are the same”

---

## Final output includes

* comment_id
* user_id
* username
* time
* preview

---

# Types of JOINs

## 1. INNER JOIN (most common)

```text id="j10"
INNER JOIN
```

### Meaning

* Only matching records appear
* If no match exists → row is excluded

---

## 2. LEFT JOIN

```text id="j11"
LEFT JOIN
```

### Meaning

* Keep ALL rows from the left table
* Even if no match in right table

Example:

* All comments always appear
* Even if user is missing

---

## 3. RIGHT JOIN

```text id="j12"
RIGHT JOIN
```

### Meaning

* Keep all rows from the right table
* Less commonly used

---

## 4. FULL OUTER JOIN

```text id="j13"
FULL OUTER JOIN
```

### Meaning

* Includes EVERYTHING from both tables
* Matching and non-matching rows

---

## 5. CROSS JOIN

```text id="j14"
CROSS JOIN
```

### Meaning

* Every row combined with every other row

### Example

If:

* Table A = 3 rows
* Table B = 2 rows

Result:

```text id="j15"
3 × 2 = 6 rows
```

### Warning

This can explode in size quickly:

* 1,000 × 1,000 = 1,000,000 rows

---

## 6. SELF JOIN

```text id="j16"
SELECT *
FROM employees e1
JOIN employees e2
ON e1.manager_id = e2.employee_id;
```

### Meaning

* A table joins with itself
* Used for hierarchical data (managers, trees, etc.)

---

# NATURAL JOIN (Automatic matching)

```sql id="j17"
SELECT
  comment_id,
  comments.user_id,
  users.username,
  time,
  LEFT(comment, 20) AS preview
FROM
  comments
NATURAL INNER JOIN users
WHERE
  board_id = 39;
```

---

## What it does

Automatically joins tables using columns with the same name.

---

## How it works

If both tables have:

```text id="j18"
user_id
```

PostgreSQL automatically assumes:

```text id="j19"
comments.user_id = users.user_id
```

---

## Why it's rarely used

* Hidden behavior
* Less explicit
* Harder to debug

So developers prefer:

```text id="j20"
INNER JOIN ... ON ...
```

---

# Key Concepts Summary

* JOIN combines data from multiple tables
* `INNER JOIN` → only matching rows
* `LEFT JOIN` → keep all left table rows
* `RIGHT JOIN` → keep all right table rows
* `FULL OUTER JOIN` → keep everything
* `CROSS JOIN` → every combination of rows
* `SELF JOIN` → table joins itself
* `ON` defines how tables match
* `LEFT()` extracts part of a string
* `AS` renames output columns

# NOTE: 

It's necessary to prefix column names with the table name when the column name exists in multiple tables being joined and would be ambiguous. For example, if both the comments and users tables have a user_id column, you need to specify comments.user_id or users.user_id. If a column name is unique across all tables in the query, the prefix is optional but often included for clarity.

---

# One-Sentence Summary

**JOINs allow you to combine related data across multiple tables using matching keys, enabling richer queries like showing usernames instead of just IDs in results.**

# Foreign Keys in PostgreSQL — Simple Summary

Foreign keys are used to create **relationships between tables** in a database. They connect data in one table to data in another, ensuring everything stays consistent.

---

# What is a Foreign Key?

A foreign key is a column in one table that **references the primary key of another table**.

Example idea:

* A comment belongs to a user
* So `comments.user_id` points to `users.user_id`

---

## Simple Relationship Example

```text id="fk1"
users table              comments table
-----------              ----------------
user_id (PK)  <------    user_id (FK)
username                 comment
email                    time
```

---

# Why Foreign Keys Matter

## Main Benefit: Data Integrity

Foreign keys ensure that relationships between tables are always valid.

---

## What this prevents

Without foreign keys, this could happen:

```text id="fk2"
comments table
-------------------------
user_id | comment
9999    | "Hello world"
```

But user_id `9999` might NOT exist in the users table.

---

## With foreign keys

PostgreSQL will reject invalid data:

> “You cannot insert a comment for a user that doesn’t exist.”

---

## Why this is important

* Keeps data consistent
* Prevents broken relationships
* Avoids orphan records (data pointing to nothing)

---

# Trade-off

Foreign keys add:

* Slight performance overhead

But they give:

* Strong data safety
* Automatic validation
* Better long-term reliability

---

# Full Schema Example (Users, Boards, Comments)

```sql id="schema1"
CREATE TABLE users (
  user_id INTEGER PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
  username VARCHAR (25) UNIQUE NOT NULL,
  email VARCHAR (50) UNIQUE NOT NULL,
  full_name VARCHAR (100) NOT NULL,
  last_login TIMESTAMP,
  created_on TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP
);
```

---

## What this table does

Stores user information.

### Key ideas:

* `user_id` → primary key
* `username`, `email` → unique identifiers
* `created_on` → auto timestamp

---

```sql id="schema2"
CREATE TABLE boards (
  board_id INTEGER PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
  board_name VARCHAR (50) UNIQUE NOT NULL,
  board_description TEXT NOT NULL
);
```

---

## What this table does

Stores discussion boards.

### New concept: `TEXT`

* Like `VARCHAR`
* But for long, unlimited text
* Used for descriptions, posts, etc.

---

```sql id="schema3"
CREATE TABLE comments (
  comment_id INTEGER PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
  user_id INT REFERENCES users(user_id) ON DELETE CASCADE,
  board_id INT REFERENCES boards(board_id) ON DELETE CASCADE,
  comment TEXT NOT NULL,
  time TIMESTAMP
);
```

---

## What this table does

Stores comments made by users on boards.

---

# Foreign Key Breakdown

## 1. `user_id INT REFERENCES users(user_id)`

```text id="fk3"
user_id INT REFERENCES users(user_id)
```

### What it means

* `INT` → stored as an integer
* `REFERENCES users(user_id)` → links to users table

### Relationship

Each comment belongs to a user.

---

## 2. `board_id INT REFERENCES boards(board_id)`

```text id="fk4"
board_id INT REFERENCES boards(board_id)
```

### What it means

Each comment belongs to a board.

---

# ON DELETE CASCADE

```text id="fk5"
ON DELETE CASCADE
```

---

## What it does

If the referenced row is deleted, PostgreSQL automatically deletes related rows.

---

## Example

If a user is deleted:

```text id="fk6"
users table
user_id = 5 (deleted)
```

Then all their comments are automatically deleted:

```text id="fk7"
comments table
user_id = 5 → removed automatically
```

---

## Why this is useful

* Prevents orphaned comments
* Keeps database clean automatically
* Reduces manual cleanup work

---

# Other DELETE Options

## 1. `ON DELETE NO ACTION` (default)

* Prevents deletion if related data exists
* Forces manual cleanup first

---

## 2. `ON DELETE SET NULL`

```text id="fk8"
user_id = NULL
```

* Keeps comment
* Removes relationship to user

---

# Key Concepts Summary

* Foreign keys link one table to another
* They enforce valid relationships between data
* Prevent invalid or orphaned records
* `REFERENCES` defines the relationship
* `ON DELETE CASCADE` auto-deletes related rows
* `TEXT` is used for long, unbounded strings
* Schema design is critical for data integrity

---

# One-Sentence Summary

**Foreign keys connect tables together by linking primary keys across tables, enforcing data integrity and ensuring relationships stay valid even when data is updated or deleted.**
